# `/weekly-review` 使用指南

> 每週回顧與知識蒸餾 — 你的週日 AI 助手

---

## 快速上手

在 Claude Code 中輸入：

```bash
/weekly-review
```

AI 會自動完成六件事：匯總本週數據 → 生成週記 → 編譯知識 → 歸檔過期內容 → 審查專案 → 檢查 Wiki 健康。

**建議**：每週日傍晚運行，作為本週收尾儀式。

---

## 命令參數

| 命令 | 用途 | 適合場景 |
|------|------|----------|
| `/weekly-review` | 完整六步驟 | 週日例行回顧 |
| `/weekly-review --dry-run` | 僅預覽，不寫入 | 首次使用前測試 |
| `/weekly-review --quick` | 僅生成週記 | 平日快速檢查 |
| `/weekly-review --step 3` | 僅執行步驟 3 | 中斷後恢復 |

---

## 六步驟詳解

### Step 1：本週數據匯總 📊

AI 掃描你本週 7 天的日記，自動統計：

- ✅ 完成了多少任務
- 📥 Inbox 進出量
- 📝 編譯了多少 Wiki 頁面
- 📅 哪幾天沒寫日記
- 💡 收集了哪些閃念（標記值得升級為專案的）

### Step 2：生成週記 📝

在 `Periodic/weekly/` 下創建結構化週記，包含 8 個區塊：

```
📊 本週數據    — 數字儀表板
🎯 專案進展    — 每個活躍專案的快照
📚 知識積累    — 本週新建的 Wiki 頁面
🔮 下週優先級  — AI 建議的 Top 3
💡 閃念回顧    — 值得跟進的想法
🔍 Wiki 健康   — 系統質量報告
🤔 反思        — 留給你手動填寫
```

> ⚠️ `## 🤔 反思` 區塊永遠不會被 AI 覆蓋，放心寫。

### Step 3：批量知識編譯 🔨

把本週收集到 `raw/` 的資料自動編譯成結構化 Wiki 頁面。

- 只處理本週新入的資料
- 每個主題最多 10 個文件（控制成本）
- 超出部分排入下週

### Step 4：過期內容歸檔 🗂️

自動找出 > 14 天前的臨時內容，移入 `4 Archives/`。

**安全機制**：
- 被其他筆記引用的文件不會被歸檔
- 日記中的引用不算（日記引用視為過期引用）
- 標記 `keep: true` 的文件永不歸檔

### Step 5：專案狀態審查 🔍

| 檢測項 | 動作 |
|--------|------|
| 截止日期已過 | 列入週記提醒你確認 |
| > 30 天無更新 | 標記 ⚠️ 停滯 |
| 活躍專案 | 自動摘要進展 + 建議下週行動 |

### Step 6：Wiki 健康檢查 🩺

快速系統體檢：

- 🔗 死鏈（指向不存在文件的連結）
- 🏝️ 孤立頁（沒有被引用的頁面）
- ⚡ 矛盾標記（不同來源的衝突觀點）
- 📉 低置信度頁面（需要人工複審）

## 典型場景

### 場景一：週日例行回顧

```bash
/weekly-review
```

→ 完整運行，約需 2-5 分鐘（取決於 raw/ 資料量）

### 場景二：週三快速檢查

```bash
/weekly-review --quick
```

→ 只生成週記初稿，不編譯不歸檔，30 秒內完成

### 場景三：中斷後恢復

假設上次在 Step 3 中斷了（知識編譯太花時間）：

```bash
/weekly-review --step 4
```

→ 從歸檔步驟繼續，跳過已完成的 1-3

### 場景四：先看好不好再執行

```bash
/weekly-review --dry-run
```

→ 看看會處理什麼，確認後再正式運行

## FAQ

### Q: 會刪除我的文件嗎？
**不會。** 歸檔 = 移動到 `4 Archives/`，不是刪除。所有內容都可找回。

### Q: 如果本週沒寫日記怎麼辦？
Step 1 會跳過數據匯總，在週記中標記「本週無日記記錄」。其餘步驟正常執行。

### Q: 週三可以跑嗎？
可以，但建議週日跑。Step 3 只會處理本週一到當天的 raw 資料，週日補跑一次即可。

### Q: 和 `/lint` 有什麼區別？
`/lint` 是全系統深度檢查（建議每月一次），`/weekly-review` 的 Step 6 是快速版本（建議每週一次）。

### Q: 如何防止某個文件被歸檔？
在 frontmatter 中添加 `keep: true`：

```markdown
---
keep: true
---
```

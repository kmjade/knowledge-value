## 1. 設計哲學

### 1.1 三條核心公理

#### 公理一：單一 Vault，永不分裂

> 所有內容進入同一個 Obsidian Vault。多 Vault 切換的摩擦是偽問題——真正的解是在單一 Vault 內建立明確的隔離邊界。

- 一個 Vault，全部 Markdown，全部對 AI Agent 可訪問
- 權限控制通過 AI 規則實現，而非物理分割

#### 公理二：AI 是編譯器 + 構建者

> AI 從「對話夥伴」升級為「知識庫構建者」。人類定義主題（一句話/一個詞），AI 完成結構化構建（14+ 文件、DDC 分類、Git 提交）。

v1.0 定義：「人類維護 raw/，AI 維護 wiki/」
v1.1 擴展：「AI 直接構建完整 KB——從 DDC 分類、MOC 導航到章節內容」

#### 公理三：信息有生命周期，知識庫有結構標準

> 不是所有信息都值得永久存儲。但值得存儲的，必須遵循統一的結構標準。

- DDC 提供普世分類框架
- 標準化 KB 結構確保一致性
- AI 負責維護，人類負責消費和方向指引

### 1.2 設計原則

| 原則 | v1.0 | v1.1 擴展 |
|------|------|-----------|
| **關注點分離** | raw/ vs wiki/ | + KB 結構層（00-MOCs, 01-09, ram/） |
| **標準化優先** | Frontmatter | + DDC 分類 + KB 結構模板 |
| **溯源優先** | sources/ | + `source:` frontmatter + wiki 鏈接 |
| **AI 可構建** | /wiki-compile | + 單詞命令觸發完整 KB 構建 |
| **故障安全** | Markdown 人類可讀 | Git 版本控制可回滾 |

## 4. DDC 知識分類體系

### 4.1 分類原則

| 原則 | 說明 |
|------|------|
| **主分類** | 000–900 十大類（DDC 標準） |
| **子分類** | 精確到小數點後（如 332.6, 619.1） |
| **命名** | `<DDC編號> <英文/繁體名稱>` |
| **自定義** | 標準 DDC 未覆蓋領域使用用戶自定義編號 |
| **語言** | 目錄名用英文 + 繁體中文、內容用繁體中文 |

### 4.2 分類碼規則

| 類型 | 格式 | 例子 |
|------|------|------|
| 標準 DDC | `DDD 名稱` | `610 医学健康` |
| DDC 小數 | `DDD.D 名稱` | `332.6-投资与交易` |
| 自定義 | `DDD 名稱` (非標準編號) | `689 业余手工` |
| 中文經典 | `DDD.D(XXX) 名稱` | `619.1(093)-1 黄帝内经` |

### 4.3 導航系統

```
3 Resources/
├── _META-INDEX.md          ← 全局導航入口
├── 3 Resources.md          ← Resources 總索引
└── [DDD]/[DDD].md          ← 每個 DDC KB 的 MOC 入口
```

## 7. LLM-Wiki 子庫模式

### 7.1 模式概述

LLM-Wiki 是嵌入在 DDC KB 內的可選子模式，用於需要 AI 持續編譯的領域：

```
[DDC KB]/
├── ... (標準 KB 結構)
├── raw/              原始素材（人類/AI 收集）
│   ├── articles/     網頁剪藏
│   ├── papers/       論文
│   ├── books/        書摘
│   └── conversations/ AI 對話
├── wiki/             AI 編譯產物
│   ├── index.md      知識索引
│   ├── log.md        編譯日誌
│   ├── concepts/     概念頁
│   ├── entities/     實體頁
│   └── sources/      來源頁
└── CLAUDE.md         子庫 Schema 定義
```

### 7.2 使用場景

| 場景 | 是否使用 LLM-Wiki | 理由 |
|------|:---:|------|
| 快速構建新領域 KB | ❌ | 直接生成標準章節更快 |
| 持續積累的研究領域 | ✅ | raw/ 持續增長，定期編譯 |
| 需要 AI 輔助發現關聯 | ✅ | wiki/ 的關係網絡有價值 |
| 一次性完成的參考 KB | ❌ | 標準結構已足夠 |

### 7.3 編譯規則

1. **數據脫敏**：個人數據不進入 wiki/
2. **raw/ 只讀**：AI 不修改原始資料
3. **來源必須**：所有 wiki 頁面必須標註 sources
4. **鏈接優先**：使用 `[[]]` 建立知識連接
5. **增量編譯**：只處理 `compiled: true` 為 false 的文件

### 7.4 Wiki 頁面模板

#### 概念頁面

```markdown
---
aliases: [別名]
created: YYYY-MM-DD
type: concept
status: draft | reviewed | evergreen
---

# [概念名稱]

## 定義
[一句話定義]

## 核心原理
[詳細解釋]

## 實踐應用
[具體案例]

## 相關概念
- [[related-concept]]

## Sources
- [[source-file]]
```

#### 實體頁面

```markdown
---
aliases: [別名]
created: YYYY-MM-DD
type: entity
entity_type: person | organization | product | tool
---

# [實體名稱]

## 基本信息
- 類型:
- 領域:

## 說明
[描述]

## 相關概念
- [[related-concept]]

## Sources
- [[source-file]]
```

## 10. 多語言策略

### 10.1 語言分層

| 層級 | 語言 | 說明 |
|------|------|------|
| **目錄名** | English + 繁體中文 | `610 医学健康` |
| **標題** | 繁體中文 + English | `# 外匯交易 Forex Trading` |
| **內容** | 繁體中文 | 全文章節正文 |
| **技術術語** | English（保留原文） | MT4, RSI, MQL4, BPU |
| **章節標題** | 雙語 | `## 快速導航 Quick Navigation` |
| **程式碼** | English | MQL/Python/C++ 範例 |

### 10.2 標點規則

- 繁體中文使用全形標點：，。：；？！「」『』
- 英文使用半形標點：, . : ; ? !
- 中英混排時英文前後加空格

## 11. Git 工作流

### 11.1 Commit 規範

## 🎯 問題卡结构详解

### 1. 📋 問題基本資訊

| 字段 | 說明 | 示例 |
|------|------|------|
| 問題标题 | 简洁明确的問題描述 | "API 接口響應超时" |
| 問題类型 | 問題所属类别 | 技術問題、流程問題、业务問題 |
| 優先級 | 問題的紧急程度 | 高/中/低 |
| 創建時間 | 問題發現的時間 | 2026-01-27 |
| 负责人 | 問題處理的主要责任人 | 张三 |

**使用技巧**：
- 問題标题應該包含關鍵資訊，便于搜尋
- 優先級设定要考虑影响范围和紧急程度
- 明确负责人可以避免责任不清

---

### 2. 🎯 問題描述

#### 問題描述
詳細描述問題的症状和现象，包括：
- 发生了什么异常
- 正常情况下應該是什么样
- 問題的具体表现

#### 影响范围
評估問題对各方的影响：
- [ ] 个人 - 只影响自己
- [ ] 團隊 - 影响整个團隊
- [ ] 使用者 - 影响最终使用者
- [ ] 系統 - 影响系統整體運行

#### 预期行为 vs 实际行为
對比正常和异常的情况，明确問題所在。

#### 复现步骤
如果問題可以复现，詳細記錄步骤：
1. 第一步操作
2. 第二步操作
3. 第三步操作

**使用技巧**：
- 尽量量化問題（如：響應時間从 100ms 增加到 5s）
- 記錄問題发生的環境和条件
- 如果問題随机发生，記錄出现频率

### 4. 🛠️ 解決方案探索

#### 方案對比框架

| 方案 | 思路 | 优点 | 缺点 | 实施难度 | 时效性 |
|------|------|------|------|----------|--------|
| 方案一 | ... | ... | ... | 简单/中等/困难 | 快速/中等/慢速 |
| 方案二 | ... | ... | ... | 简单/中等/困难 | 快速/中等/慢速 |
| 方案三 | ... | ... | ... | 简单/中等/困难 | 快速/中等/慢速 |

**使用技巧**：
- 至少探索 2-3 个方案
- 优先考虑能从根本上解決問題的方案
- 平衡短期缓解和长期根治
- 評估方案的风险和副作用

---

### 5. ✅ 实施方案

#### 選擇的方案
記錄最终選擇的方案编号和選擇理由。

#### 实施步骤
詳細的实施步骤，包括：
1. 具体操作步骤
2. 需要的資源和支持
3. 涉及的人员和协调事項

#### 驗證
# 方法
- 驗證結果：[ ] 成功 [ ] 失败
- 如果失败，返回探索方案階段

**使用技巧**：
- 实施步骤要具体可执行
- 考虑回滚方案，以防实施失败
- 驗證要充分，确保問題真正解決

# 分析

# 方法
通過连续问"为什么"来深入挖掘問題的根本原因：

**問題根本原因**：...

- 为什么？（第一层原因）
- 为什么？（第二层原因）
- 为什么？（第三层原因）
- 为什么？（第四层原因）
- 为什么？（第五层根本原因）

**示例**：
```
問題：網站崩溃

为什么1？資料庫連接池耗尽
为什么2？查詢语句没有優化，导致大量慢查詢
为什么3？没有对查詢效能進行監控
为什么4：開發團隊缺乏效能測試流程
为什么5：專案初期没有建立完整的品質保障体系
根本原因：缺乏完整的品質保障体系
```

**使用技巧**：
- 不要在表层原因停止
- 每一个"为什么"都要有证据支持
- 根本原因通常是系統性或流程性的問題
- 解決根本原因才能真正预防問題再次发生

## 💡 最佳實踐

### 1. 及时創建
- 發現問題立即創建問題卡
- 不要等到問題恶化才開始記錄
- 創建時間越早，記錄的資訊越准确

### 2. 客观描述
- 使用客观的语言，避免情绪化表达
- 基于事实和數據，而非主观判斷
- 区分"症状"和"原因"

# 分析
- 不要满足于表层原因
# 方法
# 分析

### 4. 多方探索
- 不要局限于第一个想到的方案
- 比较至少 2-3 个方案
- 考虑短期缓解和长期根治

### 5. 彻底驗證
- 驗證要充分，不能只測試一次
- 考虑边界情况和异常場景
- 監控問題是否真正解決

### 6. 系統總結
- 經驗要具体可操作
- 预防措施要有明确责任人
# 知識庫

### 7. 定期回顧
- 定期回顧已關閉的問題卡
- 檢查预防措施是否有效
- 从歷史問題中提取模式

## PARA 簡介

### 什麼是 PARA？

PARA 是一個資訊組織系統，將所有資訊分為四個類別：

| 字母 | 英文 | 中文 | 核心問題 |
|------|------|------|----------|
| **P** | Projects | 項目 | 需要什麼行動？ |
| **A** | Areas | 領域 | 需要維持什麼標準？ |
| **R** | Resources | 資源 | 對什麼主題有興趣？ |
| **A** | Archives | 歸檔 | 什麼不再需要了？ |

### 為什麼使用 PARA？

- **簡單**：只有四個分類
- **實用**：按行動價值組織
# 工作流
- **可擴展**：從個人到團隊都適用

### PARA vs 傳統分類

| 傳統分類 | PARA |
|----------|------|
| 按主題（如：學習、工作） | 按行動價值 |
| 固定結構 | 動態優先順序 |
| 容易堆積灰塵 | 定期清理歸檔 |

## 四個分類

### 1. Projects（項目）

**定義：** 有明確目標和期限的短期任務

**特點：**
- 有「完成」狀態
- 需要行動才能推進
- 有截止日期
- 通常持續幾天到幾個月

**判斷標準：**
```
這個任務是否有：
✓ 明確的目標？
✓ 截止日期？
✓ 完成狀態（是/否）？
```

**範例：**
```
✓ 完成季度報告（截止：3月31日）
✓ 規劃家庭旅行（截止：6月）
✓ 撰寫部落格文章系列
✓ 準備產品發布會
✓ 學習 Python 基礎（目標：1個月）
```

**非範例：**
```
✗ 提升寫作能力（這是 Area，不是 Project）
✗ 閱讀更多書籍（這是 Resource）
✗ 保持健康（這是 Area）
```

---

### 2. Areas（領域）

**定義：** 需要持續維護的長期責任

**特點：**
- 沒有「完成」狀態
- 需要定期關注
- 有標準需要維持
- 持續一輩子

**判斷標準：**
```
這個領域是否：
✓ 需要長期維護？
✓ 有標準要維持？
✓ 永遠不會真正「完成」？
```

**範例：**
```
# 管理
✓ 財務規劃
✓ 人際關係
✓ 職業發展
✓ 居住環境
✓ 個人成長
```

**建議的核心領域清單：**

| 領域 | 檢查點 |
|------|--------|
| **健康** | 運動、飲食、睡眠、體檢 |
| **財務** | 預算、儲蓄、保險、投資 |
| **人際** | 家人、朋友、同事 |
| **職業** | 技能、績效、發展 |
| **生活** | 居家、交通、休閒 |
| **學習** | 知識、技能、興趣 |

---

### 3. Resources（資源）

**定義：** 有興趣的主題和參考資料

**特點：**
- 沒有時效壓力
- 可以自由探索
- 未來可能用到
- 純粹的興趣或參考

**判斷標準：**
```
這個資訊是否：
✓ 對某個主題有興趣？
✓ 未來可能用到？
✓ 只是收集，不需立即行動？
```

**範例：**
```
✓ 機器學習研究
✓ 投資理財文章
✓ 旅行靈感
# 收藏
✓ 設計資源
✓ 有趣的Podcast
```

---

### 4. Archives（歸檔）

**定義：** 不再活躍但需要保留的內容

**特點：**
- 不需要日常關注
- 保留歷史記錄
- 可以查詢
- 佔用認知空間最少

**歸檔時機：**
- 項目已完成或取消
- 領域已不再相關
- 資源已過時
- 資訊不再需要

## 快速開始

### 步驟 1：建立資料夾結構

在 Obsidian  vault 根目錄建立：

```
PARA/
├── Projects/
├── Areas/
├── Resources/
└── Archive/
```

### 步驟 2：建立模板

建立 `Templates/PARA/` 資料夾，新增以下模板：

```markdown
- [[]]
```

```markdown
- [[]]
```

# 整理

1. 導出所有筆記標題
2. 逐一分類：
   - 這個需要行動嗎？ → Projects
   - 這是長期責任嗎？ → Areas
   - 這是興趣主題嗎？ → Resources
   - 這不再需要了？ → Archive
3. 移動到對應資料夾

### 步驟 4：設定每日檢視

建議每天/每週：
- [ ] 檢查 Projects 進度
# 更新
- [ ] 新增新 Resources
- [ ] 處理收件匣

# 工作流

**早晨（5分鐘）**
1. 打開 `/para-workflow`
2. 檢查今天的 Projects
3. 確認優先順序

**工作時（按情境）**
- 需要專注 → 做 Projects
- 零碎時間 → 處理 Areas
- 輕鬆探索 → 瀏覽 Resources

**晚間（5分鐘）**
1. 記錄新的想法/任務
# 更新
3. 規劃明天的工作

## 歸檔策略

### 歸檔時機

| 類別 | 歸檔條件 |
|------|----------|
| **Projects** | 已完成 / 已取消 / 擱置 > 3 個月 |
| **Areas** | 已不再相關（如換工作、搬家） |
| **Resources** | 已過時（如舊技術、已放棄的興趣） |

### 歸檔資料夾結構

```
PARA/
├── Archive/
│   ├── 2024/
│   │   ├── 2024-01/
│   │   ├── 2024-02/
│   │   └── ...
│   │
│   ├── Projects/
│   ├── Areas/
│   └── Resources/
│
├── Projects/
├── Areas/
└── Resources/
```

### 歸檔命名規範

```
YYYY-MM-DD_名稱_狀態
```

範例：
```
2024-01-26_網站改版_Completed
2024-02-15_日本旅行_Cancelled
2024-03-01_舊專案_Purged
```

### 狀態標籤

| 標籤 | 意義 |
|------|------|
| `_Completed` | 已完成 |
| `_Cancelled` | 已取消 |
| `_On-Hold` | 已擱置 |
| `_Merged` | 已合併 |
| `_Purged` | 已清理 |

## 常見問題

### Q1: 東西該放在 Projects 還是 Resources？

**問自己：**
- 需要行動才能推進？ → Projects
- 只是收集參考？ → Resources

**範例：**
- 「完成 Python 課程」→ Project（有行動）
- 「Python 學習資源」→ Resource（純參考）

---

### Q2: Areas 數量多少合適？

**建議：5-10 個核心領域**

太多會：
- 無法專注
- 認知負擔過大
- 難以維護

太少會：
- 遺漏重要領域
- 分類過於粗糙

**核心領域參考：**
1. 健康
2. 財務
3. 人際
4. 職業
5. 生活環境
6. 個人成長

---

### Q3: 多久歸檔一次？

**建議：**
- 項目完成後**立即歸檔**
- 每週檢視時**批量歸檔**
- 每季**深度清理**

---

### Q4: 可以用標籤代替資料夾嗎？

**可以，但建議：**

| 方式 | 優點 | 缺點 |
|------|------|------|
| 資料夾 | 視覺清晰、易移動 | 嵌套複雜 |
| 標籤 | 靈活、跨類別 | 難以維護、雜亂 |

**混合方案：**
- 主分類用資料夾
- 子分類用標籤

```
PARA/
├── Projects/              # 主分類（資料夾）
│   └── #project/work/     # 子分類（標籤）
├── Areas/                 # 主分類
│   └── #area/health/      # 子分類
└── Resources/             # 主分類
```

---

### Q5: 跨類別的內容怎麼辦？

**選項 1：放在最相關的類別**
- 職業相關的學習 → Areas/Career

**選項 2：建立連結參考**
```
在 Projects/新功能 中：
相關資源：[[Resources/相關主題]]
```

**選項 3：建立聚合頁面**
```markdown
# 聚合：跨類別主題

## Projects
- [[Project A]]

## Areas
- [[Area B]]

## Resources
- [[Resource C]]
```

---

### Q6: 擱置的項目怎麼處理？

**選項 1：保留在 Projects，標記為 On-Hold**
```markdown
status: on-hold
reason: 等待外部回應
```

**選項 2：移動到 Areas（如果是長期責任）**
- 例如：「找工作」→ Area/Career

**選項 3：歸檔**
- 超過 3 個月無進展 → 歸檔

---

### Q7: 怎麼處理參考連結？

**建議：放在 Resources**
- 文章連結
- 影片筆記
- 書籍筆記
- 研究資料

**不要放在：**
- Projects（除非是項目所需）
- Areas（除非與領域直接相關）

---

## 最佳實踐

### 1. 立即分類

收到資訊**當下**就決定類別，不要等到「待會」。

### 2. 保持簡單

不要過度複雜化結構。四個類別已經足夠。

### 3. 定期檢視

系統需要維護才能運作。建立檢視習慣。

### 4. 勇於歸檔

不要舍不得移動內容。活躍與歸檔分離。

### 5. 建立連結

善用 wiki links 連接相關內容：
```markdown
相關：[[Projects/相關項目]]
參考：[[Resources/相關資源]]
```

### 6. 使用模板

一致性節省時間。模板確保結構完整。

### 7. 命名規範

# 排序
```markdown
YYYY-MM-DD_專案名稱
領域_專案名稱
```

### 8. 設定優先順序

# 排序
- 高優先權在前
- 截止日期近的在前

### 9. 記錄上下文

在筆記中記錄「為什麼」：
```markdown
<!-- 為什麼創建這個筆記？ -->
<!-- 這個資訊來自哪裡？ -->
```

### 10. 持續優化

每季檢視 PARA 結構，調整不適用的部分。

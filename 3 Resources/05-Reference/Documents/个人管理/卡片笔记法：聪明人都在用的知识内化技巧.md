## 二、为什么卡片筆記法有效？

### 2.1 原因一：强制你理解

写卡片时有一个规则：**必须用自己的话写。**

不是複製貼上，是「翻译」成自己能懂的表达。

這個過程会强制你思考：「我真的理解了吗？」

**能用自己的话说清楚，才叫真懂。**

### 2.2 原因二：原子化便于复用

一张卡片只写一个点，意味着它是独立的。

text

```
場景1：写文章需要讲「缓存穿透」   → 直接拿出「缓存穿透」这张卡片      
場景2：面试需要复习 Redis   → 把所有 Redis 相關的卡片拉出来     
場景3：做技術方案需要參考   → 找「解決方案」类型的卡片
```

**一张好卡片，可以被用無数次。**

# 知識

卡片之间可以互相連結。

当你把「缓存穿透」和「布隆过滤器」连起来时，你会發現新的关系。

当你把「限流算法」和「熔断機制」放在一起时，你会产生新的思考。

## 三、一张好卡片长什么样？

### 3.1 卡片的四个要素

text

`┌─────────────────────────────────────────────┐   
│              一张好卡片的结构               │   
├─────────────────────────────────────────────┤   
│                                             │   
│  1. 标题（这张卡片讲什么）                  │   
│     → 一眼看懂主題                          │   
│                                             │   
│  2. 一句话總結                              │   
│     → 用自己的话，说清楚核心                │   
│                                             │   
│  3. 詳細內容                                │   
│     → 展开说，但不超过 300 字               │   
│                                             │   
│  4. 关联卡片                                │   
│     → 和哪些卡片有关系                      │   
│                                             │   
└─────────────────────────────────────────────┘`

### 3.2 真实示例

```Markdown
# 布隆过滤器      
## 一句话總結
   > 用很小的空間快速判斷「一定不存在」，代价是可能误判「存在」。      
## 詳細內容   
- 本质：位数组 + 多個哈希函数   
- 新增：用 k 个哈希算出 k 个位置，全部置 1   
- 查詢：k 个位置都是 1 → 可能存在；有一个 0 → 肯定不存在   
- 特點：有假阳性，無假阴性      
## 我的理解   
# 效率
「肯定不存在」這個结论很有用。      
## 关联卡片   
- [[缓存穿透]]   
- [[位图算法]]
```

### 3.3 好卡片的三个標準

怎么判斷卡片写得好不好？

```text
標準1：原子化   ─────────────   只讲一个点，不贪多   能拆就拆，宁小勿大      
標準2：自足性   ─────────────   不依赖上下文也能看懂   别人看到也能理解      
標準3：有「我」   ─────────────   有自己的话、自己的理解   不是複製貼上
```

**不符合这三条的，要么拆分，要么重写。**

## 四、怎么写卡片？我的流程

### 4.1 阅读时：標記「卡片点」

看书或文章时，遇到值得记的点，先標記下来。

text

```
📍 这里是一个概念 → 可以写成「概念卡」   
# 方法
📍 这里颠覆了我的认知 → 可以写成「洞见卡」
```

**不是什么都记，只记有價值的「卡片点」。**

### 4.2 標記后：问自己三个問題

拿到一个卡片点后，问自己：

text

```
Q1：這個点的核心是什么？（一句话）   
Q2：我用自己的话怎么说？   
# 知識
```

回答完这三个問題，卡片就写完了。

### 4.3 写完后：建立連接

每张新卡片写完，问自己：

**这张卡片和哪些已有卡片相關？**

用 `[[双向連結]]` 连起来。



```Markdown
## 关联卡片   
- [[缓存穿透]] ← 布隆过滤器可以解決穿透問題   
- [[位图算法]] ← 布隆过滤器基于位图   
- [[概率型數據结构]] ← 布隆过滤器是其中一种
- 
```

**連接越多，这张卡片越有價值。**

## 五、不同类型的卡片

我把卡片分成几种类型：

### 5.1 概念卡

解釋「是什么」。

```Markdown
# 缓存穿透      
## 一句话總結   
> 查詢一个不存在的數據，缓存和資料庫都查不到，每次都打到資料庫。      
## 詳細內容   
...
```

# 方法

解釋「怎么做」。

```Markdown
# 缓存穿透解決方案      
## 一句话總結   
# 方法
> ## 詳細內容   
# 方法
> - 原理：...   
> - 优点：...      
# 方法
> - 原理：...   
> - 注意：...`
```
### 5.3 對比卡

解釋「有什么区别」。



```Markdown
# RDB vs AOF      
## 一句话總結   
> RDB 是快照，恢復快但可能丢數據；AOF 是日志，數據安全但檔案大。      
## 對比表格   
| 维度 | RDB | AOF |   
|------|-----|-----|   
| 原理 | 定时快照 | 記錄每条命令 |   
| 恢復速度 | 快 | 慢 |   
| 數據安全 | 可能遺失 | 更安全 |
```

### 5.4 洞见卡

記錄你的「顿悟」。



```Markdown
# 【洞见】缓存的本质      
## 一句话總結   
> 缓存的本质不是「存數據」，而是「用空間换時間」。      
> ## 我的思考   
> 以前我以为缓存就是把數據放到更快的地方。   
> 后来發現，缓存的核心是一种权衡：   
> - 用記憶體空間换取访问速度   
> - 用一致性风险换取可用性   
> - 用系統复杂度换取效能提升      
> 每次設計缓存方案，都要想清楚這個权衡。
```

## 七、常见問題

### Q1：每张卡片應該多长？

**一屏以内。**

手机一屏能看完最好，最长不超过 300 字。

太长了說明可以拆分。

### Q2：什么內容值得写成卡片？

问自己：**這個点以后可能用到吗？**

可能用到 → 写卡片  
只是背景資訊 → 不用写

**宁缺毋滥。**

### Q3：卡片太多了怎么找？

# 方法

```text
1. 搜尋關鍵词   
2. 通過標籤篩選   
3. 从 MOC（內容地图）导航
```

只要標籤打得好，找起来很快。

### Q4：已有的长筆記怎么办？

不用一次性全部拆分。

用到的时候再拆。

**渐进式遷移，不要追求完美。**

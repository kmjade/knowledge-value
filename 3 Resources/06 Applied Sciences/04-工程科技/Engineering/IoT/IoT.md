# 物联网 (IoT)

> **物联网 = 通过传感器 + 通信网络 + 计算平台，将物理世界连接到数字世界的技术体系。**
>
> 它是数字孪生、智能制造和智慧城市的基础设施——没有 IoT，数字世界就是"盲"的。

IoT 不是单一技术，而是一个**技术栈**：从物理层的 MEMS 传感器，到网络层的 WiFi/5G/LoRa，到平台层的设备管理和规则引擎，再到应用层的预测性维护和智慧城市。这个栈上每一层的选择都会影响其他层——一个"便宜好用"的 IoT 方案需要四层协同设计。

## IoT 四层架构

```
┌────────────────────────────────────┐
│         应用层                      │  智慧工厂/城市/农业/医疗/车联网
├────────────────────────────────────┤
│         平台层                      │  设备管理、规则引擎、数据存储、分析
│         (AWS IoT/Azure/EMQX)       │
├────────────────────────────────────┤
│         网络层                      │  WiFi/BLE/Zigbee/LoRa/5G/NB-IoT
│         (MQTT/CoAP/HTTP)           │
├────────────────────────────────────┤
│         感知层                      │  传感器、执行器、RFID、摄像头
│         (MEMS/温度/压力/振动)        │
└────────────────────────────────────┘
```

> **核心洞察**: IoT 的瓶颈通常在**网络层和感知层之间**——传感器数据上不来，或者上来了但协议不兼容。平台层和应用层再强大，没有可靠的数据管道也毫无用处。

## 消费 IoT vs 工业 IoT (IIoT)

这是 IoT 最根本的分野——相同技术栈，完全不同的约束条件：

| 维度 | 消费 IoT | 工业 IoT (IIoT) |
|------|----------|-----------------|
| 故障后果 | 不便 | **停产 / 安全事故 / 环境灾难** |
| 可靠性 | 可接受偶尔断连 | 99.9%+ 可用性 |
| 环境 | 舒适的室内 | -40~85°C、粉尘、振动、防爆 |
| 生命周期 | 2-5 年 | 10-30 年 |
| 协议 | WiFi、BLE、Matter | Modbus、Profinet、EtherCAT、OPC UA |
| 安全 | 数据隐私 | 人身安全 + 生产安全 + 国家安全 |
| 设备规模 | 10-100/户 | 1,000-100,000/厂 |

> **IIoT 的核心矛盾**: 工业设备寿命 10-30 年，但 IoT 技术迭代周期 2-3 年。OPC UA 和协议网关的存在意义就在这里——在新老设备之间建桥。

## 关键技术决策框架

### 通信协议选型

选择 IoT 通信协议的本质是四个维度的权衡：

| 协议 | 距离 | 带宽 | 功耗 | 成本 | 典型场景 |
|------|------|------|------|------|----------|
| **WiFi 6/7** | 短 (~50m) | 极高 (Gbps) | 高 | 低 | 视频监控、AR |
| **BLE 5.x** | 短 (~100m) | 低 (2Mbps) | 极低 | 极低 | 传感器标签、信标 |
| **Zigbee/Thread** | 短 (~100m) | 低 (250kbps) | 极低 | 低 | 智能家居 Mesh |
| **LoRaWAN** | 长 (~10km) | 极低 (50kbps) | 极低 | 低 | 农业、抄表 |
| **NB-IoT** | 长 (运营商) | 低 (250kbps) | 低 | 中 | 城市基础设施 |
| **5G** | 长 (运营商) | 极高 (Gbps) | 高 | 高 | 自动驾驶、远程手术 |
| **有线 (EtherCAT)** | 极短 | 极高 | — | 中 | 运动控制、机器人 |

> **没有"最好"的协议——只有最适合约束条件的协议。** 一个农业 IoT 方案用 WiFi = 每隔 50 米放一个 AP = 荒谬。一个工厂运动控制用 LoRa = 延迟太大 = 危险。

### IoT 安全：不是附加项，是设计约束

IoT 安全的独特性在于：**设备资源极度受限（MCU + 64KB RAM），不能用传统的 TLS + 证书方案。** 但安全需求不因资源受限而降低。

| 层级 | 威胁 | 对策 |
|------|------|------|
| **设备** | 固件篡改、克隆 | 安全启动 + 硬件唯一密钥 (PUF) |
| **通信** | 窃听、中间人 | DTLS/OSCORE (轻量 TLS for constrained devices) |
| **平台** | 未授权访问、DDoS | X.509 证书 + 设备影子 + 权限最小化 |
| **数据** | 隐私泄露 | 边缘预处理 + 匿名化 |

> Mirai (2016) 的教训：几十万台被入侵的 IoT 摄像头和路由器，靠**默认密码**就打垮了半个美国的互联网。IoT 安全的最大漏洞往往不是加密算法，而是**出厂默认密码和永不更新的固件**。

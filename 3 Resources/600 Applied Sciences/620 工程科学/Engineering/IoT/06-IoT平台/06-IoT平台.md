# 06-IoT 平台

> IoT 平台是连接设备与应用的中间层——设备接入、数据存储、规则引擎、可视化和分析。没有平台，每个 IoT 项目都要从轮子开始造。

## 平台核心能力

| 能力 | 说明 |
|------|------|
| **设备管理** | 注册、认证、配置、OTA 固件升级 |
| **数据接入** | MQTT/HTTP/CoAP 协议支持 |
| **规则引擎** | 温度>90→发告警→调用API |
| **数据存储** | 时序数据库——高效存储时间序列 |
| **可视化** | 自定义 Dashboard |
| **分析** | 聚合、异常检测、ML 集成 |
| **API 开放** | RESTful API 供上层应用调用 |

## 主流平台对比

| 平台 | 类型 | 优势 | 适合 |
|------|------|------|------|
| **AWS IoT Core** | 公有云 | 生态最全、设备影子 | 全球部署 |
| **Azure IoT Hub** | 公有云 | 企业级、DPS | 微软生态 |
| **阿里云 IoT** | 公有云 | 中国最大 | 国内企业 |
| **EMQX** | 开源 Broker | 百万连接、轻量 | 自建平台 |
| **ThingsBoard** | 开源平台 | 规则引擎+看板 | 中小项目 |
| **Ignition** | 工业SCADA | 无限许可 | 工业自动化 |

## 设备影子 (Device Shadow)

AWS IoT 的核心概念——每个设备在云端有一个 JSON 文档代表其期望状态和上报状态：

```json
{
  "state": {
    "desired": {"temperature": 23.0},  // 用户想设的温度
    "reported": {"temperature": 25.0}  // 设备上报的实际温度
  }
}
```

### 为什么需要设备影子？

| 场景 | 没有影子 | 有影子 |
|------|----------|--------|
| 设备离线 | 无法发送指令 | 指令存入影子→设备上线后同步 |
| 多应用控制 | 冲突 | 影子是单一数据源 |
| 状态查询 | 需要轮询设备 | 读影子即可 |

## 规则引擎

```
IF 温度 > 90°C → 发送告警 → 存入 DynamoDB → 触发 Lambda
```

| 动作 | 例子 |
|------|------|
| 存储 | 写入时序数据库、S3 |
| 通知 | SMS、邮件、MQTT 消息 |
| 调用 | Lambda 函数、HTTP API |
| 控制 | 下发指令到设备 |

## 自建 vs 云平台

| 维度 | 云平台 | 自建开源 |
|------|--------|----------|
| 初始成本 | 低 | 高 (需要团队) |
| 运营成本 | 按量付费→大项目贵 | 固定成本 |
| 灵活性 | 受限于平台 | 完全自由 |
| 数据控制 | 数据在第三方 | 数据在自己手里 |
| 适合 | 初创、标准需求 | 大企业、定制需求 |

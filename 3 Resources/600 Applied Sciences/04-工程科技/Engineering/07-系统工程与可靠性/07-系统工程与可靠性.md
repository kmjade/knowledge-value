# 07-系统工程与可靠性

> 系统工程是管理复杂性的学科——当系统由成千上万个组件构成、需要多个团队协作时，如何确保最终产品按预期工作。可靠性是系统工程的核心目标之一。

## 系统工程 V 模型

```
需求分析 ←──────────────────→ 系统验证与确认
    ↓                               ↑
架构设计 ←──────────────────→     集成测试
    ↓                               ↑
子系统设计 ←──────────────→      子系统测试
    ↓                               ↑
组件设计 ←──────────────→        组件测试
    ↓                               ↑
      ────── 实现/制造 ──────────→
```

左侧：分解（从整体到零件）。右侧：验证（从零件到整体）。

## 需求工程

| 类型 | 含义 | 例子 |
|------|------|------|
| **功能需求** | 系统必须做什么 | "电梯必须在 60 秒内到达任何楼层" |
| **非功能需求** | 系统要多好 | "电梯故障率 < 0.001%，噪音 < 55dB" |
| **约束** | 必须满足的外部条件 | "符合 ASME A17.1 电梯安全标准" |

> 需求错误是所有工程错误中修正成本最高的——在需求阶段修正一个错误需要 \$1，到了运行阶段需要 \$1000+。

## 可靠性工程

### 浴盆曲线

```
故障率
  ↑
  │ \                           /
  │  \  早期失效    随机失效    /  耗损失效
  │   \  (缺陷)   (恒定速率)   /  (老化)
  │    \_____________________/
  └────────────────────────────→ 时间
```

### 关键指标

| 指标 | 含义 |
|------|------|
| **MTBF** (Mean Time Between Failures) | 平均故障间隔——可修复系统 |
| **MTTF** (Mean Time To Failure) | 平均失效时间——不可修复系统 |
| **可用性** | MTBF / (MTBF + MTTR)——系统可用的时间比例 |
| **失效率 λ** | 单位时间内的失效概率 |

### 冗余配置

| 类型 | 工作方式 | 可靠性提升 |
|------|----------|-----------|
| 冷备份 | 备用不工作，主件坏了才启动 | 高 |
| 热备份 | 主备同时工作 | 中等 |
| 三重冗余 (TMR) | 三个同时工作，投票表决 | 极高（航天、核电） |

## FMEA (故障模式与影响分析)

系统性地分析每个组件可能的故障模式、后果和检测方法。

| 要素 | 含义 |
|------|------|
| 故障模式 | 组件可能怎样坏？ |
| 故障影响 | 对系统的后果是什么？ |
| 严重度 (S) | 后果有多严重？1-10 |
| 发生概率 (O) | 发生的可能性？1-10 |
| 检测能力 (D) | 故障能被多快发现？1-10 |
| **RPN** | = S × O × D，越高越需要优先处理 |

## 故障树分析 (FTA)

从上到下：从一个不希望发生的顶事件出发，逆向分析所有可能导致它的原因。

```
       "发动机无法启动" (顶事件)
              OR
      ┌───────┴────────┐
   无燃料          无点火
      OR             AND
  ┌───┴───┐     ┌───┴────┐
油泵坏  管路漏  火花塞坏  ECU故障
```

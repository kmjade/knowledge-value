# 04-Matter 安全模型

> Matter 的安全设计原则：每个设备都有唯一身份、所有通信加密、只有被证明"真"的设备才能加入网络。

## 安全架构

```
┌─────────────────────────────────────────────┐
│           应用层: 端到端加密消息               │
├─────────────────────────────────────────────┤
│           会话层: 单播/组播会话密钥            │
├─────────────────────────────────────────────┤
│           传输层: TLS/DTLS                   │
├─────────────────────────────────────────────┤
│           网络层: IPsec (Thread)             │
└─────────────────────────────────────────────┘
```

## 设备身份：DAC

每个 Matter 设备出厂时带有一张**设备认证证书 (DAC)**——设备的"身份证"：

| 证书内容 | 说明 |
|----------|------|
| 设备唯一 ID | 全球唯一的标识 |
| 厂商 ID (VID) | 谁制造的 |
| 产品 ID (PID) | 哪款产品 |
| 公钥 | 对应设备内部的私钥 |
| CSA 签名 | CSA 证明"这是一台合法的 Matter 设备" |

## 配网流程

```
1. 用户扫描设备上的二维码
2. 手机和 Matter 设备建立 BLE 连接
3. PAKE (密码认证密钥交换) —— 安全地交换密钥
4. 手机把 WiFi/Thread 凭证发给设备
5. 设备用凭证接入网络
6. 配网完成——设备加入 Fabric
```

### 为什么用二维码而不是手动输入密码？

- 二维码包含配网密码和鉴别信息
- 防止配网过程中的中间人攻击
- 比手动输入 12 位密码快且准

## 分布式合规账本 (DCL)

Matter 不使用中心化的设备数据库——而是用区块链式的分布式账本：

| 功能 | 说明 |
|------|------|
| 查询 | 这个设备是什么？谁造的？ |
| 验证 | DAC 证书有效吗？ |
| 更新 | 固件更新可用吗？ |

> 分布式账本保证了即使 CSA 的服务器宕机，已部署的设备仍然可以正常配网和运行。

## 安全特性总结

| 特性 | 实现 |
|------|------|
| 设备身份 | X.509 证书 (DAC) |
| 通信加密 | AES-CCM |
| 消息完整性 | HMAC |
| 防重放 | 消息序列号+时间戳 |
| 固件安全 | 签名验证+安全启动要求 |
| 隐私 | 设备间通信不泄露到云端 |

# 07-IoT 安全

> IoT 安全不是可选项——Mirai 僵尸网络 (2016) 用百万 IoT 设备攻击 DNS，导致半个美国断网。安全从芯片设计开始，贯穿整个生命周期。

## 威胁模型

| 攻击面 | 威胁 |
|--------|------|
| **设备** | 物理篡改、固件逆向、侧信道攻击 |
| **通信** | 窃听、中间人攻击、重放攻击 |
| **云端** | API 漏洞、数据泄露、账号劫持 |
| **供应链** | 预装后门、假冒芯片 |

## 设备安全

### 安全启动

```
芯片 ROM → 验证 Bootloader 签名 → 验证固件签名 → 启动
   │              ✗                     ✗
   信任根        拒绝                  拒绝
```

### 硬件安全

| 技术 | 说明 |
|------|------|
| **Secure Element (SE)** | 独立安全芯片——存储密钥、执行加密，防物理攻击 |
| **TEE (可信执行环境)** | CPU 内的安全区域——ARM TrustZone |
| **PUF (物理不可克隆函数)** | 芯片制造差异→唯一指纹→不可克隆 |
| **防篡改** | 开盖检测→自动擦除密钥 |

## 通信安全

| 层级 | 措施 |
|------|------|
| **设备认证** | X.509 证书、预共享密钥 (PSK) |
| **传输加密** | TLS 1.3 (TCP), DTLS (UDP) |
| **消息签名** | 每一条 MQTT 消息都可被签名验证 |
| **重放保护** | 时间戳+序列号 |

## 设备身份生命周期

```
出厂: 注入唯一证书 → 激活: 云端注册 → 运行: 定期认证 → 报废: 撤销证书
```

### AWS IoT 的认证方式

| 方式 | 适合 |
|------|------|
| X.509 证书 | 设备出厂预置——推荐 |
| IAM 角色 | 运行在 AWS 上的设备 |
| Cognito | 用户+设备混合场景 |

## 固件安全

| 阶段 | 措施 |
|------|------|
| 开发 | 签名固件、代码审计、SBOM |
| 分发 | 加密传输、校验签名 |
| 更新 (OTA) | 差分更新、断点续传、回滚保护 |
| 运行 | 最小权限、只读文件系统 |

## 零信任在 IoT 中

| 原则 | IoT 实现 |
|------|----------|
| 永不信任，始终验证 | 每次通信都认证 |
| 最小权限 | 设备只能访问必要的资源 |
| 持续监控 | 异常行为实时检测 |
| 假设已被攻破 | 隔离——一个设备被攻破不影响其他 |

## Mirai 的教训

2016 年 Mirai 僵尸网络攻击——核心原因：

| 问题 | 教训 |
|------|------|
| 默认密码未修改 | 出厂必须强制修改密码 |
| 固件无法更新 | OTA 是必备功能 |
| 无异常检测 | 设备发大量 DNS 请求→异常 |

> 今天 IoT 厂商仍然在犯 Mirai 时代的错误。CSA 的 Matter 标准和安全认证正在改善这一点。

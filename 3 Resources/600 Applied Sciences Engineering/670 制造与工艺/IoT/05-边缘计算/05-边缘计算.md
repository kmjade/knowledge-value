# 05-边缘计算

> 边缘计算在数据产生的地方（或附近）处理数据——不是把所有数据都送到云端。减少延迟、节省带宽、提高可靠性。

## 云-边-端架构

```
     云端: 大模型训练、全局分析、长期存储
      ↑↓ (选择性数据)
     边缘: 实时处理、协议转换、本地决策
      ↑↓ (全部数据)
     终端: 传感器、执行器
```

## 为什么需要边缘计算？

| 驱动力 | 说明 |
|--------|------|
| **延迟** | 自动驾驶不能等 100ms 云往返——必须毫秒级响应 |
| **带宽** | 一个工厂每天产生 TB 级数据——全传云端不现实 |
| **可靠性** | 断网时边缘仍能独立工作 |
| **隐私** | 敏感数据（医疗、军工）不出本地 |
| **成本** | 云端存储和传输费用——边缘过滤掉 90% 无用数据 |

## 边缘网关

| 功能 | 说明 |
|------|------|
| **协议转换** | Modbus/Profinet→MQTT/HTTP |
| **数据预处理** | 去噪、重采样、特征提取、压缩 |
| **本地规则引擎** | 温度>90°C→立即停止→不依赖云端 |
| **离线缓存** | 网络中断→本地存储→恢复后补传 |
| **运行 ML 推理** | TensorFlow Lite/TinyML——在边缘做预测 |
| **安全** | 防火墙、加密、设备认证 |

## 边缘 ML (TinyML)

在微控制器上运行机器学习：

| 框架 | 特点 |
|------|------|
| **TensorFlow Lite Micro** | Google——最主流 |
| **Edge Impulse** | 低代码 ML for MCU |
| **MicroTVM** | Apache——灵活 |

### 典型 TinyML 应用

| 应用 | 模型 | 硬件 |
|------|------|------|
| 电机故障检测 | 振动异常分类 | Cortex-M4, <1mW |
| 关键词唤醒 | 语音分类 | Cortex-M7 |
| 人员计数 | 简单视觉模型 | ESP32-S3 + 摄像头 |

> TinyML 的意义：让每个传感器都变成"智能传感器"——在采集数据的同一芯片上判断是否异常，只在上报时才唤醒通信。

## 边缘 vs 雾 vs 云计算

| 层级 | 位置 | 延迟 | 计算力 | 典型硬件 |
|------|------|------|--------|----------|
| **终端** | 传感器/MCU | μs | 极低 | Cortex-M |
| **边缘** | 网关/嵌入式PC | ms | 中 | Cortex-A/Atom |
| **雾** | 本地服务器 | 10ms | 高 | Xeon/GPU |
| **云** | 数据中心 | 50-200ms | 极高 | 集群 |

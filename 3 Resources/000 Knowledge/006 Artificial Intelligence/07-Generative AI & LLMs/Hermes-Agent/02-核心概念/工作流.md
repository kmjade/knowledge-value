# 工作流

> [!summary] 概述
> Hermes Agent 的常见工作流模式。

---

## 单智能体工作流

最基础的模式——一个 Hermes 实例处理一个任务：

```
用户任务 → Hermes 推理 → 工具调用 → 结果 → 最终回复
```

```bash
hermes chat -q "分析 ~/project 的代码结构"   # 单次
hermes                                        # 多轮交互
```

---

## 并行多智能体（tmux）

同时运行多个独立的 Hermes 实例：

```bash
# Agent A：后端开发
tmux new-session -d -s backend -x 120 -y 40 'hermes -w'
sleep 8 && tmux send-keys -t backend '构建用户管理 REST API' Enter

# Agent B：前端开发
tmux new-session -d -s frontend -x 120 -y 40 'hermes -w'
sleep 8 && tmux send-keys -t frontend '构建用户管理 React Dashboard' Enter

# 检查进度
tmux capture-pane -t backend -p | tail -30
tmux capture-pane -t frontend -p | tail -30
```

`-w` (worktree) 模式使用独立 git 工作树，避免冲突。

## 子智能体委派（delegate_task）

在会话中使用 `delegate_task` 并行处理子任务：

```
用户: 研究项目需要同时调研三个方向

Hermes 使用 delegate_task 并行委派：
- 子任务 A：调研 React 状态管理方案
- 子任务 B：调研 Python Web 框架
- 子任务 C：调研数据库选型
→ 汇总结果给用户
```

| 特性 | delegate_task | tmux 多开 |
|------|--------------|----------|
| 隔离 | 独立上下文，共享进程 | 完全独立进程 |
| 时长 | 分钟级（受父会话限制） | 小时/天 |
| 交互 | 否 | 是（PTY） |
| 适用 | 快速并行研究/分析 | 长时间自主任务 |

## 定时自动化（Cron）

```bash
# 每小时健康检查
hermes cron create "every 1h" "检查 Docker 容器状态，异常服务重启"

# 每日报告
hermes cron create "0 9 * * *" "生成昨日系统使用报告"

# 管理
hermes cron list        # 查看
hermes cron pause ID    # 暂停
hermes cron run ID      # 立即触发
```

---

## 多 Profile 隔离

```bash
# 创建工作 Profile
hermes profile create work

# 使用不同 Profile
hermes --profile work     # 工作专用
hermes                     # 个人使用（default）
```

每个 Profile 有独立的配置、技能、记忆和会话。

---

## 网关多平台

```bash
hermes gateway setup   # 配置 Telegram/Discord 等
hermes gateway run     # 启动

# 同一智能体同时在 CLI、Telegram、Discord 可用
```

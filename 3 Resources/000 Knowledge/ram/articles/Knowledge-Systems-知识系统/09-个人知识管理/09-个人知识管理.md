# Personal Knowledge Management (PKM)

> 本章有两层身份：一是 KOS 谱系的终点——从分类法到 AI 的九章旅程在此落地；二是**本 vault 的系统架构文档**——记录这个 Obsidian 知识库的设计原则、信息流、类型系统和演进路线。

## PKM 方法论速览

| 方法 | 核心 | 代表工具 |
|------|------|----------|
| **Zettelkasten** | 原子笔记 + 双向链接 | Obsidian |
| **PARA** | 项目/领域/资源/归档 | 文件夹结构 |
| **GTD** | 清空大脑 → 执行 | Todoist |

本 vault 采用 **PARA + LLM-Wiki 融合架构**，在此基础上吸收 Zettelkasten 的原子化思想和 GTD 的行动导向。

## 二、设计原则

### 原则 1: 人机分工（Human-AI Boundary）

```
人类独占 → raw/     (原始资料，AI 只读)
混合维护 → PARA 层   (人类主导，AI 辅助)
AI 独占  → wiki/    (编译产物，人类只读)
```

这条边界防止 AI 污染一手资料，同时让 AI 在编译层充分发挥。

### 原则 2: 分层不重复（Layered, Not Duplicated）

同一知识在不同层有不同形态，但不复制内容：

```
raw/     → 原文摘要 + 个人批注    (reference)
wiki/    → 结构化概念 + 关系图    (evergreen)
outputs/ → 文章、决策、创作      (ephemeral)
```

### 原则 3: 渐进结构化（Progressive Structuring）

笔记不要求首次写就完美。结构随理解加深而生长：

```
捕获 (快速) → 分拣 (归类) → 链接 (建立关系) → 编译 (提炼概念) → 表达 (输出)
```

### 原则 4: 链接优先于分类（Links Over Folders）

文件夹是物理位置，链接是语义关系。当一个笔记属于多个类别时：

- **第一步**: 放入最主要的文件夹
- **第二步**: 用 `[[]]` 建立跨类别的语义链接
- **第三步**: 用 MOC 聚合分散的内容

### 原则 5: 可审计（Auditability）

每个知识单元必须可溯源：

- wiki 页面必须包含 `## Sources` 指向 `raw/` 原文
- 编译操作记录到 `AI-Log/compile-log.md`
- 分拣操作记录到 `AI-Log/triage-log.md`

## 三、信息流架构

### 端到端路径

```
                    信息输入层
┌─────────────────────────────────────────────────────┐
│  网页剪藏 → 0 Inbox/  或  raw/articles/              │
│  闪念笔记 → 0 Inbox/  或  Daily Notes                │
│  对话/会议 → raw/conversations/                      │
│                                                      │
│  /triage ──→ 分拣决策 ──→ 记录到 triage-log.md       │
└───────────────────────┬─────────────────────────────┘
                        ↓
                    行动管理层 (PARA)
┌─────────────────────────────────────────────────────┐
│  1 Projects/  ← 有截止日期的行动                     │
│  2 Areas/     ← 持续责任领域                         │
│  3 Resources/ ← 知识资源库                           │
│  4 Archives/  ← 已完成/过时内容                      │
│                                                      │
│  人类主导写入，AI 辅助整理和检索                      │
└───────────────────────┬─────────────────────────────┘
                        ↓
                    知识编译层 (LLM-Wiki)
┌─────────────────────────────────────────────────────┐
│  /wiki-compile ──→ 提取概念/实体/关系                │
│                                                      │
│  wiki/concepts/   ← 概念页 (如 [[Algorithmic-Trading]])│
│  wiki/entities/   ← 实体页 (如 [[MetaTrader]])       │
│  wiki/sources/    ← 溯源页                            │
│  wiki/index.md    ← 知识索引                          │
│                                                      │
│  AI 独占写入，人类通过 outputs/ 消费                   │
└───────────────────────┬─────────────────────────────┘
                        ↓
                    表达产出层
┌─────────────────────────────────────────────────────┐
│  outputs/  ← 基于 wiki 生成的制品                     │
│  文章、报告、决策笔记、演讲、项目方案                   │
│                                                      │
│  人类主导，AI 辅助生成                                │
└─────────────────────────────────────────────────────┘
```

### 各阶段工具

| 阶段 | 动作 | 工具/命令 | 产出 |
|------|------|-----------|------|
| **Capture** | 快速捕获 | Web Clipper, 闪念笔记, Daily Notes | `0 Inbox/` 条目 |
| **Triage** | 分拣决策 | `/triage` | 归类 + 标记 triaged |
| **Organize** | 分类链接 | 手动 + Agent 辅助 | PARA 结构 + wikilinks |
| **Compile** | 知识编译 | `/wiki-compile` | wiki/ 页面 |
| **Express** | 创作输出 | 手动 + Agent 辅助 | outputs/ 制品 |
| **Review** | 健康检查 | `/lint` (规划中) | 结构审计报告 |

## 四、类型系统

### Note 类型定义

| 类型 | 说明 | 存放位置 | Schema 标记 |
|------|------|----------|-------------|
| **inbox** | 待分拣条目 | `0 Inbox/` | `triaged: false` |
| **project** | 有截止日期的任务集 | `1 Projects/` | `type: project` |
| **area** | 持续责任领域 | `2 Areas/` | `type: area` |
| **resource** | 知识资源 | `3 Resources/` | `type: resource` |
| **archive** | 归档内容 | `4 Archives/` | `type: archive` |
| **raw** | 原始资料（人类独占） | `[子库]/raw/` | `type: source` |
| **wiki-concept** | AI 编译的概念页 | `[子库]/wiki/concepts/` | `type: concept` |
| **wiki-entity** | AI 编译的实体页 | `[子库]/wiki/entities/` | `type: entity` |
| **wiki-source** | AI 编译的溯源页 | `[子库]/wiki/sources/` | `type: source` |
| **moc** | 内容地图（导航页） | `00-MOCs/` 或子库根 | `type: moc` |
| **daily** | 日记 | `Daily Notes/` | 自动生成 |
| **session-log** | AI 会话记录 | `AI-Log/sessions/` | `type: session-log` |
| **output** | 基于 wiki 的制品 | `[子库]/outputs/` | `type: output` |

### Wiki 页面标准 Frontmatter

```yaml
---
aliases:
  - 别名1
  - 别名2
created: YYYY-MM-DD
updated: YYYY-MM-DD
type: concept | entity | source | moc | output
topic: ai-ml | people | finance | productivity | epistemology | knowledge-systems
status: draft | reviewed | evergreen
sources:
  - "[[raw/articles/some-article]]"
---
```

### 原始资料标准 Frontmatter

```yaml
---
created: YYYY-MM-DD
source: https://example.com 或 书籍/对话描述
type: article | paper | book | conversation | note
tags:
  - tag1
  - tag2
---
```

### 分拣标记

```yaml
---
triaged: true
triaged_at: 2026-05-28T14:30:00
triaged_to: 3 Resources/ai-ml/raw/articles/
---
```

## 五、知识拓扑（链接网络）

### 链接类型

| 链接类型 | 语法 | 语义 | 示例 |
|----------|------|------|------|
| **层级链接** | `[[parent]]` | "属于"分类 | `[[02-Classification体系]]` ← 09 隶属于此 |
| **概念链接** | `[[concept]]` | "涉及"某概念 | `[[Algorithmic-Trading]]` |
| **来源链接** | `[[raw/xxx]]` | "源自"某资料 | `[[raw/articles/mt4-guide]]` |
| **交叉链接** | `[[other-topic/note]]` | "相关"跨领域 | `[[../../03-Productivity/PARA]]` |
| **索引链接** | MOC 中的列表 | "汇总"分散内容 | MOC 页面的聚合链接 |

### 链接密度标准

- **Wiki 概念页**: 至少 2 个 outgoing links（关联概念 + 来源）
- **MOC 页**: 聚合 5-20 个子页面链接
- **Raw 资料**: 可无 outgoing links（单向被引用）
- **孤立笔记**: 零 incoming + 零 outgoing — 需要分拣或归档

## 六、维护协议

### 例行节奏

| 频率 | 动作 | 时间预算 | 触发 |
|------|------|----------|------|
| **每日** | Inbox 清零（≤5 条堆积） | 5-10 min | 打开 vault 时 |
| **每周** | `/triage` 批量分拣 + Git push | 15-30 min | 周末 |
| **每月** | 检查孤立笔记 + 清理重复标签 | 30 min | 月初 |
| **每季度** | `/wiki-compile` 批量编译 + 架构 review | 1-2 h | 季末 |

### 维护判断标准

| 信号 | 含义 | 行动 |
|------|------|------|
| Inbox > 20 条 | 分拣管道堵塞 | 立即 triage |
| 孤立笔记 > 总笔记 10% | 链接网络稀疏 | 批量建立链接或归档 |
| wiki/ 页面无 Sources | 溯源链断裂 | 补充来源 |
| 同一概念 2+ wiki 页 | 重复编译 | 合并 + 修复链接 |
| raw/ 被 AI 修改 | 越界 | 回滚 + 审计 |

## 九、Obsidian 即 KOS

回到本章开头——你的 Obsidian vault 不只是一个笔记工具。它是一套**运行中的知识组织系统**：

| 你做的 | KOS 等价 |
|--------|----------|
| PARA 文件夹 | 分类法 (DDC-style) |
| `tags:` frontmatter | 受控词表 / 标签系统 |
| `aliases:` frontmatter | 叙词表入口词 |
| `[[wikilink]]` | 关联索引 / 语义关系 |
| YAML `type:` `topic:` | 本体属性 |
| MOC 导航页 | 分类导航 / 地图 |
| Claude 子库 schema | 本体约束 |
| Dataview 查询 | 推理 / 跨库检索 |
| `/triage` `/wiki-compile` | 知识管理流程 |
| Claudian Agent | AI 驱动的知识引擎 |

**每一行都是一个正在运作的 KOS 组件。** 前八章的理论不在书本里——它们就在你的文件夹、标签、链接和编译日志里运行着。

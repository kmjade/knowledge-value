# 04 — Agent 框架與運維

## 1. Agent 框架總覽

### 1.1 全局架構

```
                        ┌────────────────────────────────────┐
                        │     CLAUDE.md                      │
                        │    (系統總綱)                      │
                        └────────────────────────────────────┘
                                   │ 引用配置
              ┌────────────────────┼────────────────────┐
              │                    │                    │
        ┌─────▼──────┐      ┌─────▼──────┐      ┌─────▼──────┐
        │   Skills   │      │   Hooks    │      │  Schemas   │
        │            │      │            │      │            │
        │ /triage    │      │SessionStart│      │CLAUDE.md   │
        │ /wiki-     │      │PostToolUse │      │(子庫)      │
        │   compile  │      │            │      │            │
        │ /context   │      │            │      │            │
        │ /lint      │      │            │      │            │
        └────────────┘      └────────────┘      └────────────┘
               │                    │                    │
               └────────────────────┼────────────────────┘
                                    │
                        ┌───────────▼───────────┐
                        │     文件系統           │
                        │  (Markdown 文件)       │
                        └───────────────────────┘
```

### 1.2 CLAUDE.md — 系統總綱

**文件**: `CLAUDE.md`
**角色**: 系統級配置文件

**覆蓋章節**:

| 章節 | 內容 | 用途 |
|------|------|------|
| Vault 總攬 | 整體架構圖 | 全局認知 |
| 目錄職責速查 | 8 個目錄的職責與權限 | 路由決策 |
| 核心規則 | 5 條核心原則及其違反後果 | 行為約束 |
| 信息處理流程 | 4 階段流程 | 分診與編譯 |
| Frontmatter 標準 | 統一格式 | 數據規範 |
| 對話協議 | 開始/結束流程 | 溝通規則 |
| 禁止行為 | 4 條絕對禁止 | 安全邊界 |

### 1.3 Skills — 技能目錄

四個核心 Skills 位於 `.claude/skills/`：

| Skill | 指令 | 職責 |
|-------|------|------|
| Triage | `/triage` | Inbox 文件分類與路由 |
| Wiki-Compile | `/wiki-compile` | 原始資料 → Wiki 頁面 |
| Context | `/context` | 會話狀態彙報 |
| Lint | `/lint` | 系統檢查與修復 |

### 1.4 Hooks — 自動化觸發

**文件**: `.claude/hooks/hooks.json`

| Hook 名稱 | 觸發時機 | 功能 |
|-----------|----------|------|
| SessionStart | 每次會話開始 | 顯示 Vault 狀態 |
| PostToolUse | Write/Edit 後 | 記錄文件修改 |

### 1.5 Schemas — 子庫定義

每個 Wiki 子庫對應一個 `CLAUDE.md` Schema 文件：

```
3 Resources/[topic]/CLAUDE.md
├── 目錄結構約定
├── 核心概念列表
├── 頁面模板
├── 術語表
└── 使用示例
```

## 2. 安全約束

### 2.1 AI 權限邊界

```
允許操作:
✅ 讀取 Vault 任何文件
✅ 寫入 wiki/ 目錄
✅ 寫入 AI-Log/ 目錄
✅ 移動 0 Inbox/ 中的文件
✅ 在 1 Projects/ 2 Areas/ 4 Archives/ 中創建文件
✅ 創建新目錄
✅ 更新 Frontmatter

禁止操作:
❌ 修改 raw/ 目錄中的任何文件
❌ 刪除文件（除非明確確認）
❌ 修改 .obsidian/ 配置文件
❌ 一次性創建超過 10 個文件（除非明確確認）
❌ 在 wiki/ 中修改非 AI 生成的內容
❌ 暴露個人信息到 wiki/ 頁面
```

### 2.2 Git 操作規範

```
操作前自動檢查:
1. 操作前: git status 確認工作目錄乾淨
2. 操作後: git status 確認變更範圍
3. 分批提交: 避免單次提交過大
4. 危險操作: 要求用戶確認

提交規範:
- 每次 /triage 或 /wiki-compile 完成後提交
- 提交信息遵循 conventional commits 規範
- 單次提交不超過 50 個文件

遠程同步:
- push 前確保網絡通暢
- 優先使用 SSH，配置好
- token 權限要求: repo
```

### 2.3 隱私與安全

| 敏感類型 | 處理策略 |
|----------|----------|
| 個人財務信息 | 不導入 wiki/ |
| 聯繫方式 | 存儲在 people/raw/，不在 wiki/ 公開 |
| API 密鑰 | 不存儲在 Vault 中 |
| 私人對話 | 標記 sensitive: true 不編譯 |
| 醫療信息 | 不導入到 wiki/ |

### 2.4 容錯處理

```
故障場景           影響              恢復方式
──────────────────────────────────────────────────────────
Git 連接失敗        無法同步遠端      本地繼續工作
AI 不可用           知識編譯停擺      用戶仍可直接操作 Markdown
網絡不穩定          無法聯網          手動處理 Inbox
目錄意外刪除        結構破壞          從 Git 恢復
Skill 配置錯誤      工作異常          從 Git 恢復後重新部署
格式錯誤            顯示異常          運行 /lint 檢查並修復
```

## 3. 日常運維

### 3.1 會話流程

```
┌─────────────────────────────────────────────────────────────┐
│              會話開始（每次）                                │
│                                                             │
│  1. 掃描 Vault                                              │
│  2. 運行 /context --quick                                   │
│  3. 檢查 Inbox 是否有待處理                                  │
│  4. 確認當前活躍項目                                        │
└─────────────────────────────────────────────────────────────┘
                  │
                  │
┌─────────────────────────────────────────────────────────────┐
│              中間會話                                       │
│                                                             │
│  當有新信息時:                                               │
│    整理 → Inbox → /triage                                   │
│                                                             │
│  當有未編譯資料時:                                           │
│    /wiki-compile [topic]                                    │
│                                                             │
│  /context 隨時查看狀態                                       │
└─────────────────────────────────────────────────────────────┘
                  │
                  │
┌─────────────────────────────────────────────────────────────┐
│              會話結束                                       │
│                                                             │
│  1. 運行 /lint --quick                                      │
│  2. 檢查 Git 狀態                                          │
│  3. 提交變更                                               │
│  4. 推送到遠端                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 維護頻率

| 頻率 | 任務 | 工具 |
|------|------|------|
| 每天 | Inbox 分診 | `/triage` |
| 每天 | 狀態檢查 | `/context --quick` |
| 每 2-3 天 | 編譯知識 | `/wiki-compile [topic]` |
| 每週 | 快速檢查 | `/lint` |
| 每週 | 詳細檢查 | `/lint --report` |
| 每季度 | 歸檔回顧 | 檢查 4 Archives/ 內容 |
| 每年 | 戰略複盤 | 檢查並清理無價值內容 |

### 3.3 創建新子庫

創建一個新的 Wiki 子庫的標準流程：

```
1. 確定主題和覆蓋範圍
2. 創建目錄結構:
   mkdir -p "3 Resources/[topic]/{raw/{articles,papers,books,conversations},wiki/{concepts,entities,sources},outputs}"
3. 創建 CLAUDE.md (Schema 定義)
4. 創建 wiki/index.md (索引頁)
5. 創建 wiki/log.md (編譯日誌)
6. 更新 3 Resources/_META-INDEX.md
7. 添加初始 raw/ 內容
8. 運行 /wiki-compile [topic]
9. 驗證: /lint
```

### 3.4 故障排除

| 症狀 | 檢查 | 解決方案 |
|------|------|----------|
| Inbox 積壓過多 | `/context --inbox` | 運行 `/triage --scope` 分批處理 |
| Wiki 編譯失敗 | 查看 compile-log | 用 `--dry-run` 預覽問題 |
| 格式不一致 | `/lint` 檢查 | 運行 `/lint --fix` |
| Git 狀態混亂 | `git status` | 分批提交 |
| Skill 無效 | 檢查 skill.json | 確認 enabled: true |

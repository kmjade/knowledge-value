# 05 — 日常維護流程

> **規則性質**: 強制性運維規則（Rule），非建議性指南。
> **適用對象**: AI Agent（Claudian）及人類用戶。
> **更新頻率**: 每次會話開始時加載，每季度檢討修訂。

---

## 1. 每日開機儀式 (Morning)

> ⏰ 建議時間：每天首次打開 Vault 時執行
> ⏱️ 預計耗時：2-5 分鐘

### 1.1 快速狀態檢查

```
Step 1: /daily-open          ← 檢查/創建今日日記，填充活躍任務與 Inbox 狀態
Step 2: /context --quick     ← 快速狀態總覽（6 指標表）
Step 3: 閱讀今日日記          ← 確認「今日意圖」與「昨日遺留」
```

### 1.2 狀態評估矩陣

執行 `/context --quick` 後，根據以下矩陣判斷優先級：

| Inbox | raw/ 待編譯 | 距上次 lint | 判斷 | 動作 |
|:-----:|:----------:|:----------:|------|------|
| ≤5 | ≤3 | ≤7 天 | 🟢 健康 | 正常開始工作 |
| 6-19 | 4-9 | 8-14 天 | 🟡 注意 | 今日安排處理 |
| ≥20 | ≥10 | >14 天 | 🔴 積壓 | **立即處理，優先於其他工作** |

### 1.3 開機決策樹

```
/context --quick
        │
        ├─ Inbox ≥ 20？ ──── 是 ──→ 🔴 P0: 立即 /triage
        │                            ↓
        │                          繼續工作前先清積壓
        │
        ├─ 有 Git 衝突？ ─── 是 ──→ 🔴 P0: 先解決衝突
        │
        ├─ raw/ 待編譯 ≥ 10？─ 是 ──→ 🟡 P1: 今日安排 /wiki-compile
        │
        ├─ 距上次 lint > 14 天？─ 是 ──→ 🟡 P1: 今日運行 /lint
        │
        └─ 以上皆否 ─────────────→ 🟢 正常開始工作
```

### 1.4 日記檢查清單

打開今日日記後，確認以下區塊：

- [ ] `## 🌅 今日意图` — 今日最重要的 1-3 件事
- [ ] `## ✅ 今日完成` — 回顧昨日遺留
- [ ] `## 🌙 日终回顾`（昨日）— 「給 AI 的指令」是否有待執行
- [ ] 本週週記（`Periodic/weekly/YYYY/YYYY-Www.md`）— 本週目標是否更新

## 2. 工作中維護 (During Day)

> 在日常工作中觸發，非定時任務。

### 2.1 信息捕獲 → Inbox

任何新信息進入 Vault 時：

```
外部來源（Web Clipper / 手動 / AI 對話）
        │
        ▼
   0 Inbox/                         ← 統一入口
        │
        ├─ 網頁截取 → Clippings/
        ├─ 碎片想法 → 1-想法/
        ├─ 待辦事項 → 2-任務/
        ├─ 臨時筆記 → 3-筆記/
        └─ 其他     → 根目錄
```

### 2.2 觸發式 Triage

**觸發條件**（滿足任一即執行）：

| 條件 | 動作 |
|------|------|
| Inbox 根目錄新增 ≥ 3 個 .md 文件 | `/triage` |
| Clippings/ 新增 ≥ 5 個文件 | `/triage --scope clippings` |
| 手動添加了重要參考資料 | `/triage --file "name.md"` |
| 用戶主動要求 | `/triage` |

**分揀後必做**:
- [ ] 確認所有文件已路由到正確位置
- [ ] 檢查 `AI-Log/triage-log.md` 本次記錄
- [ ] 對於路由到 `raw/` 的文件，標記 `compiled: false`

### 2.3 觸發式 Wiki-Compile

**觸發條件**（滿足任一即執行）：

| 條件 | 動作 |
|------|------|
| 某 topic 的 `raw/` 新增 ≥ 3 個 `compiled: false` 文件 | `/wiki-compile [topic]` |
| 用戶主動要求 | `/wiki-compile [topic]` |
| 閱讀某主題時發現新概念未收錄 | 記錄後批次 `/wiki-compile [topic]` |

**編譯優先級**:

| 優先級 | topic | 理由 |
|:------:|------|------|
| P1 | 當前活躍項目的 topic | 即時需求 |
| P2 | 最近 7 天有新增 raw/ 的 topic | 保持更新 |
| P3 | 其他 topic | 定期批次處理 |

### 2.4 隨時可用命令

| 命令 | 用途 | 場景 |
|------|------|------|
| `/context` | 完整狀態 | 需要全面了解時 |
| `/context --inbox` | Inbox 詳情 | 準備分揀前 |
| `/context --projects` | 項目狀態 | 切換工作上下文 |
| `/lint --quick` | 快速檢查 | 修改多個文件後 |
| `git status` | Git 狀態 | 準備提交前 |

## 3. 每日關機儀式 (Evening)

> ⏰ 建議時間：每天結束工作前執行
> ⏱️ 預計耗時：3-8 分鐘

### 3.1 關機流程

```
Step 1: /lint --quick          ← 快速健康檢查（關鍵項）
Step 2: git status --short     ← 檢查變更範圍
Step 3: git add + commit       ← 提交今日變更
Step 4: 更新今日日記            ← 填寫「日終回顧」
Step 5: git push               ← 推送到遠端
```

### 3.2 Git 提交決策

| 變更文件數 | 行為 |
|:---------:|------|
| 0 | 無需提交，直接結束 |
| 1-10 | ✅ 自動提交（Conventional Commits 格式） |
| 11-20 | ⚠️ 顯示變更列表，確認後提交 |
| >20 | 🔴 強制分批，每批 ≤10 個文件 |

### 3.3 日終回顧模板

在今日日記中填寫：

```markdown
## 🌙 日终回顾
- **完成了什么**：[今日主要產出]
- **未完成**：[遺留事項]
- **明日最重要**：[明日 1-3 件優先事項]
- **给 AI 的指令**：[下次會話 AI 應優先處理的事項]
```

### 3.4 關機檢查清單

- [ ] `/lint --quick` 無嚴重錯誤
- [ ] `git status` 已確認變更範圍
- [ ] 今日變更已提交（commit message 符合 Conventional Commits）
- [ ] 已推送到 GitHub（`git push`）
- [ ] 今日日記「日終回顧」已填寫
- [ ] 如有未完成事項，已記錄到明日日記或 tasks.md

## 4. 決策閾值

> 所有閾值檢查自動化通過 `/context --summary` 執行。

### 4.1 閾值總表

| 指標 | 🟢 健康 | 🟡 注意 | 🔴 積壓 | 動作 |
|------|:------:|:------:|:------:|------|
| **Inbox 積壓** | ≤5 | 6-19 | ≥20 | `/triage` |
| **最舊 Inbox 文件** | ≤3 天 | 4-7 天 | >7 天 | 優先分揀 |
| **raw/ 待編譯** | ≤3 | 4-9 | ≥10 | `/wiki-compile` |
| **距上次 /lint** | ≤7 天 | 8-14 天 | >14 天 | `/lint` |
| **距上次 Git commit** | ≤1 天 | 2-3 天 | >3 天 | `git commit` |
| **停滯項目** | ≤14 天 | 15-30 天 | >30 天 | 考慮歸檔 |
| **未跟踪 .md 文件** | 0 | 1-5 | >5 | `/triage` |

### 4.2 優先級算法

```
P0 (立即處理):
  ├─ Inbox 積壓 ≥ 20          → /triage
  ├─ 有 Git 衝突              → 手動解決
  └─ lint 發現斷鏈 > 10       → /lint --fix 或手動修復

P1 (今日處理):
  ├─ raw/ 待編譯 ≥ 10         → /wiki-compile [topic]
  ├─ 距上次 lint > 14 天      → /lint
  ├─ 最舊 Inbox > 7 天        → /triage --file
  └─ 距上次 Git commit > 3 天 → git commit + push

P2 (本週處理):
  ├─ Inbox 積壓 6-19          → /triage（分批）
  ├─ raw/ 待編譯 4-9          → /wiki-compile（選擇性）
  ├─ 項目停滯 > 30 天         → 評估歸檔
  └─ 距上次 lint 8-14 天      → 安排 /lint

P3 (本月處理):
  └─ 季度歸檔回顧             → 檢查 4 Archives/
```

### 4.3 例外規則

| 情境 | 例外處理 |
|------|----------|
| 用戶正在進行密集創作（如寫作、編碼） | 暫緩維護，完成後批次處理 |
| Vault 正在跨設備同步 | 等待同步完成後再執行寫入操作 |
| AI 不可用（如 API 配額耗盡） | 用戶手動執行基本維護 |
| 旅行/假期期間 | 降至最低維護（僅 git push 確保同步） |

## 5. 每週維護 (Weekly)

> ⏰ 建議時間：每週日或週一執行
> ⏱️ 預計耗時：15-30 分鐘

### 5.1 每週檢查流程

```
Step 1: /weekly-review        ← 週回顧：完成事項、遺留、下週計劃
Step 2: /lint --report         ← 詳細系統檢查報告
Step 3: /wiki-compile          ← 批次編譯所有有待編譯的 topic
Step 4: 檢查停滯項目           ← 標記 > 30 天無更新的項目
Step 5: Git 整理               ← 確認所有變更已提交，清理分支
Step 6: 更新週記               ← Periodic/weekly/YYYY/YYYY-Www.md
```

### 5.2 每週檢查清單

#### 系統健康
- [ ] `/lint --report` 無 Error（Warning 可接受）
- [ ] 所有 Wikilink 有效（無斷鏈）
- [ ] 所有 `raw/` 文件的 `compiled` 狀態正確
- [ ] Wiki 子庫的 `index.md` 統計數據與實際一致

#### 知識庫
- [ ] 所有有待編譯文件的 topic 已執行 `/wiki-compile`
- [ ] 新建立的概念/實體頁面有正確的 `[[ ]]` 交叉鏈接
- [ ] Wiki 頁面 Frontmatter 完整（aliases, type, topic, sources）

#### 項目管理
- [ ] 所有 `status: active` 的項目近期有更新
- [ ] 停滯 > 30 天的項目已評估（繼續 / 歸檔）
- [ ] 過期項目（deadline 已過）已處理

#### 版本控制
- [ ] 所有變更已提交並推送到 GitHub
- [ ] 無遺留的臨時分支
- [ ] `.gitignore` 正確排除不應跟踪的文件

### 5.3 月度附加檢查

在每月最後一次每週維護中附加：

- [ ] 檢查 `4 Archives/` 是否需要清理
- [ ] 檢討閾值設定是否仍然合理
- [ ] 檢查 `.obsidian/` 插件是否有更新
- [ ] 回顧 `AI-Log/` 操作日誌，識別模式

## 6. 應急處置

### 6.1 常見問題快速處置

| 症狀 | 診斷命令 | 處置 |
|------|----------|------|
| Inbox 嚴重積壓（>50 文件） | `/context --inbox` | 分 3-5 批 `/triage --scope` |
| Wiki 編譯失敗 | 查看 `AI-Log/compile-log.md` | `/wiki-compile --dry-run` 預覽問題 |
| 大量斷鏈 | `/lint` | `/lint --fix` 自動修復，或手動逐一修復 |
| Git 無法推送 | `git status` + `git log --oneline -5` | 先 `git pull --rebase` 再 `git push` |
| 格式不一致 | `/lint` 檢查 | 對比 `03-规范标准` 修正 Frontmatter |
| 文件意外丟失 | `git log --diff-filter=D --summary` | `git checkout [commit] -- [file]` 恢復 |
| Skill 無響應 | 檢查 `.claude/skills/[name]/skill.json` | 確認 `enabled: true` |

### 6.2 Git 緊急恢復

```bash
# 情況 1：誤修改了 raw/ 文件
git checkout -- "3 Resources/[topic]/raw/[file].md"

# 情況 2：需要回退到上次提交
git reset --hard HEAD

# 情況 3：找回已刪除的文件
git log --diff-filter=D --summary          # 找到刪除記錄
git checkout [commit]~1 -- [file-path]     # 恢復文件

# 情況 4：解決合併衝突
git status                                  # 查看衝突文件
# 手動編輯衝突標記後：
git add [resolved-file]
git commit
```

### 6.3 升級流程

當自身無法解決時：

```
1. 記錄問題 → AI-Log/sessions/[date]-incident.md
   ├─ 症狀描述
   ├─ 已嘗試的處置
   └─ 當前狀態

2. 保護數據
   ├─ git stash（保存當前工作）
   └─ git push（確保遠端有最新版本）

3. 降級運行
   └─ 切換到手動模式（用戶直接編輯 Markdown）
```

## 7. 命令速查

### 7.1 核心命令鏈

```bash
# 完整維護鏈（建議順序）
/context --quick          # 1. 狀態檢查
/triage                   # 2. 分揀新信息
/wiki-compile [topic]     # 3. 編譯知識
/lint                     # 4. 健康檢查

# 快速日常
/daily-open               # 開機儀式
/context --summary        # 獲取操作建議
/lint --quick             # 快速檢查
```

### 7.2 命令參數速查

| 命令 | 常用參數 | 說明 |
|------|----------|------|
| `/context` | `--quick` | 6 指標快速總覽 |
| | `--inbox` | Inbox 積壓詳情 |
| | `--projects` | 活躍項目狀態 |
| | `--summary` | 閾值 + 操作建議 |
| `/triage` | 無參數 | 全量掃描 Inbox |
| | `--dry-run` | 預覽不執行 |
| | `--file "name.md"` | 單文件分揀 |
| | `--scope clippings` | 限定範圍 |
| `/wiki-compile` | `[topic]` | 指定 topic 編譯 |
| | `--dry-run` | 僅預覽提取結果 |
| | `--incremental` | 增量編譯（僅新文件） |
| | `--force` | 強制重新編譯 |
| `/lint` | `--quick` | 僅關鍵項 |
| | `--fix` | 自動修復 |
| | `--report` | 詳細報告 |
| `/daily-open` | 無參數 | 開機儀式 |
| `/weekly-review` | 無參數 | 週回顧 |

### 7.3 Git 常用命令

```bash
git status --short          # 簡潔狀態
git add -A                  # 暫存所有變更
git commit -m "type: desc"  # Conventional Commits
git push                    # 推送
git pull --rebase           # 拉取並變基
git log --oneline -10       # 最近 10 次提交
```

## 8. 維護日誌模板

### 8.1 每日維護記錄

在今日日記中記錄：

```markdown
## 🤖 維護記錄

| 時間 | 操作 | 結果 |
|------|------|:---:|
| HH:MM | /context --quick | 🟢 |
| HH:MM | /triage | ✅ N 文件已路由 |
| HH:MM | /wiki-compile ai-ml | ✅ +3 概念 |
| HH:MM | /lint --quick | 🟢 |
| HH:MM | git commit + push | ✅ |
```

### 8.2 異常記錄模板

在 `AI-Log/sessions/` 中創建：

```markdown
---
type: incident
date: YYYY-MM-DD
severity: low | medium | high | critical
status: resolved | investigating | escalated
---

# 異常記錄：[簡述]

## 症狀
[具體表現]

## 診斷
[排查過程與發現]

## 處置
[採取的措施]

## 結果
[處置後的狀態]

## 預防
[避免再次發生的建議]
```

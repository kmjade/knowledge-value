# 01 — 設計哲學與系統架構

## 1. 設計哲學

### 三大核心理念

#### 理念一：同一 Vault，職責分離

```
所有數據存放在同一 Obsidian Vault，避免信息碎片化。
同一 Vault = 全文 Markdown = 全部可被 Claude Code 訪問。
在 Vault 內部通過目錄邊界劃分職責，而不是分割 Vault。
```

**要點**:
- 所有知識放在同一 Vault
- 不同類型的內容通過目錄邊界劃分
- 權限控制通過 AI 規則實現，而非文件系統隔離

#### 理念二：AI 不是旁觀者，而是對話者

```
從"存儲"到"編譯"的方式轉變。
捕獲信息由 AI 負責路由（/triage）。
專業知識由 AI 負責編譯（/wiki-compile）。
結構化信息由 AI 自行完成，用戶從操作者進化為決策者。
```

**要點**:
- AI 的職責是結構化知識而非存儲
- Wiki 頁面由 AI 負責維護
- 用戶維護 raw/ 由 AI 維護 wiki/

#### 理念三：信息必須流動起來

```
知識管理系統失敗的根源是"入庫容易、維護太難"。
捕獲最容易 → 組織不太難 → 維護最難 → 到一定程度後幾乎不可能。
本系統的核心設計目標：讓維護成本趨近於零。
```

**要點**:
- 只捕獲有長期信息價值的內容
- 信息需要自動歸檔和歸類
- AI 承擔維護工作，用戶專注於創作

### 設計原則

| 原則 | 說明 |
|------|------|
| **關注點分離** | raw/ 與 wiki/ 嚴格分開，各司其職 |
| **最小權限** | AI 只在授權範圍內操作 |
| **源頭追溯** | 所有 AI 生成內容標註原始來源 |
| **漸進增強** | 系統功能按需釋放，從核心到邊緣 |
| **零鎖供應商** | 即使 AI 不可用，用戶仍可直接操作 Markdown |

## 3. 三層架構設計

### 3.1 信息捕獲層 (0 Inbox/)

**職責**: 信息臨時存儲和初步處理

**設計目標**:
- 無論什麼類型的數據都可以用最小成本放入 Inbox
- 自動分類後通過 `/triage` 自動路由到對應路徑

**子目錄結構**:

```
0 Inbox/
├── Clippings/            # 網頁截取（Obsidian Web Clipper）
├── _processed/           # 已處理文件歸檔
└── _Processed.md         # 處理記錄
```

**處理策略**:

| 內容類型 | 檢測方式 | 處理方式 |
|----------|----------|----------|
| 網頁截取 | `source` frontmatter + URL | 提取元數據 → 路由到 raw/ |
| 碎片筆記 | 短內容、無明確結構 | 智能路由 |
| 待辦事項 | 列表語義 | 追加到項目 tasks.md |
| 會議筆記 | 時間信息+與會者 | 路由到 raw/ |

### 3.2 行動組織層 (PARA)

**職責**: 以行動為中心組織信息

**PARA 分類邏輯**:

```
              ┌── 有截止日期 ──┐  ┌── 無截止日期 ──┐
              │                │  │                │
      ┌───────┴───────┐      ┌─┴────────┐
      │ 1 Projects/   │      │ 2 Areas/  │
      └───────────────┘      └──────────┘
      ┌───────┴───────┐      ┌─┴────────┐
      │ 4 Archives/   │      │ 3 Resources/ │
      └───────────────┘      └──────────┘
```

#### 1 Projects/ — 項目
- 有明確目標和截止日期的行動
- 一個項目一個文件夾
- 項目完成後遷移到 4 Archives/

#### 2 Areas/ — 領域
- 長期責任範圍，無截止日期
- 需要持續關注、定期更新

#### 3 Resources/ — 資源
- 興趣主題，參考資料
- Wiki 子庫所在目錄
- 每個子庫包含 raw/ 和 wiki/

#### 4 Archives/ — 歸檔
- 非活躍項目和領域
- 僅作為歷史保留和參考

### 3.3 知識結構層 (Wiki)

**職責**: 將原始信息轉化為結構化知識

**核心原則**:

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│          raw/                          wiki/                        │
│   ┌────────────────┐             ┌────────────────┐                 │
│   │  用戶維護      │   /wiki-    │  AI 維護       │                 │
│   │  只寫不刪      │ ─────compile────?│  可讀可查     │                 │
│   │  (由 AI)       │             │  (只讀)        │                 │
│   └────────────────┘             └────────────────┘                 │
│                                                                      │
│  用戶寫入原始資料            AI 生成結構化知識                        │
│  AI 只讀取不修改             AI 獨占寫入權限                          │
│  格式自由                     格式嚴格                                │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

**轉換流程**:

```
raw/articles/*.md
raw/papers/*.md             /wiki-compile          wiki/concepts/*.md
raw/books/*.md              ──────────────?        wiki/entities/*.md
raw/conversations/*.md                             wiki/sources/*.md
                                                   wiki/index.md (總覽)
                                                   wiki/log.md   (日誌)
```

# 09-AI应用与工具

> 从实验到产品——将 AI 模型转化为实际可用的应用系统所需的工程实践、工具链和方法论。

## ML 项目生命周期

```
实验阶段                          生产阶段
┌──────────────┐                ┌─────────────────────┐
│ 数据 → 训练 → 评估 │  ──────→ │ 部署 → 监控 → 重训练 │
└──────────────┘                └─────────────────────┘
    Jupyter                      CI/CD + 监控
    快速迭代                      稳定可靠
```

| 阶段 | 关注点 | 典型环境 |
|------|--------|----------|
| 实验 | 快速试错、模型效果 | Jupyter, Colab, 本地 GPU |
| 生产 | 稳定性、延迟、成本、监控 | Docker + K8s, 云服务 |

## 模型部署

### 部署方式

| 方式 | 说明 | 适合场景 |
|------|------|----------|
| **REST API** | 模型包装为 HTTP 服务 | 最常见的方式 |
| **流式** | Server-Sent Events (SSE) | LLM 逐 token 输出 |
| **边缘部署** | 模型部署在设备本地 | 移动端、IoT |
| **批处理** | 离线大批量推理 | 数据标注、ETL 管道 |
| **WebAssembly** | 浏览器内运行小模型 | 客户端推理 |

### LLM 推理引擎

| 引擎 | 特点 |
|------|------|
| **vLLM** | PagedAttention，高吞吐量，连续批处理 |
| **TensorRT-LLM** | NVIDIA 优化，最高性能 |
| **llama.cpp** | CPU + GPU 混合推理，量化支持好 |
| **Ollama** | 一键本地部署，适合个人使用 |
| **SGLang** | 结构化生成，编程式 LLM 调用 |

| 类别 | 工具 | 说明 |

## 训练基础设施

| 规模 | 方案 |
|------|------|
| **个人/学习** | Google Colab, single GPU (RTX 30xx/40xx), Lambda Labs |
| **小团队** | 单机多 GPU, RunPod, Vast.ai |
| **中等规模** | 云 GPU 集群 (AWS p4d, GCP TPU), 多节点 |
| **大模型预训练** | 数千 GPU 集群，需要分布式训练 (FSDP, DeepSpeed) |

### 分布式训练

| 策略 | 说明 |
|------|------|
| **数据并行 (DP)** | 每 GPU 一份模型副本，不同 batch 数据 |
| **模型并行 (MP)** | 模型层分布到不同 GPU |
| **流水线并行 (PP)** | 微批次流水线处理 |
| **张量并行 (TP)** | 单层内的参数切分 |
| **ZeRO / FSDP** | 优化器状态、梯度和参数分片（DeepSpeed, PyTorch FSDP） |

## LLM 应用的工程实践

### LLM 应用架构模式

```
用户 → 网关(限流/认证) → 编排层(LangChain/DSPy)
                            ├── LLM 调用
                            ├── 向量检索 (RAG)
                            ├── 工具调用
                            └── 缓存/回退
```

### 关键考虑

| 方面 | 实践 |
|------|------|
| **缓存** | 相同/相似查询缓存结果，节省成本 |
| **回退** | 主模型故障时切换到备用模型 |
| **限流** | 保护 API 不被滥用 |
| **评估** | 自动评估 + 人工抽查 |
| **可观测性** | 记录每次调用的 token 数、延迟、成功率 |
| **A/B 测试** | 对比不同 prompt / 模型的线上效果 |

### LLM 评估

| 方法 | 说明 |
|------|------|
| **基准评估** | MMLU, HumanEval, GSM8K 等标准基准 |
| **LLM-as-Judge** | 用更强的 LLM 评估输出质量 |
| **人工评估** | Elo 评分，A/B 对比 |
| **任务特定指标** | RAG 的忠实度和相关性、代码的通过率 |

## 成本优化

| 策略 | 节省 |
|------|------|
| 使用更小的模型处理简单任务 | 7B 模型比 70B 便宜 10-100 倍 |
| Prompt 缓存 | 复用系统 prompt，减少重复 token |
| 量化 (INT8/INT4) | 减少显存和计算 |
| 批量推理 | 合并多个请求一起处理 |
| 路由 | 简单问题用小模型，复杂问题用大模型 |

# 06-数据库系统

> 数据库系统是用于高效存储、检索和管理数据的软件系统。现代应用的"数据层"——选对数据库、用好索引、理解事务，决定系统的可靠性和性能。

## 关系数据库核心概念

### ACID 事务

| 特性 | 含义 | 破坏时的后果 |
|------|------|-------------|
| **A**tomicity (原子性) | 事务要么全部成功，要么全部失败 | 转账转出一半 |
| **C**onsistency (一致性) | 事务前后数据满足所有约束 | 数据违反业务规则 |
| **I**solation (隔离性) | 并发事务互不干扰 | 脏读、不可重复读、幻读 |
| **D**urability (持久性) | 提交的数据永不丢失 | 断电后数据消失 |

### 隔离级别

| 级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|------|------|-----------|------|------|
| **Read Uncommitted** | ❌ | ❌ | ❌ | 最快 |
| **Read Committed** | ✅ | ❌ | ❌ | 较快 |
| **Repeatable Read** | ✅ | ✅ | ❌ | 中等 (MySQL 默认) |
| **Serializable** | ✅ | ✅ | ✅ | 最慢 |

### 隔离级别的实现

| 机制 | 说明 |
|------|------|
| **MVCC (多版本并发控制)** | 每个事务看到数据的快照。PostgreSQL 和 MySQL InnoDB 的核心机制 |
| 锁 (行锁/表锁) | 显式锁定数据，阻止其他事务访问 |
| 两阶段锁 (2PL) | 严格保证可串行化 |

## 索引

### B+ 树索引（最常用）

```
          [30 | 70]            ← 内部节点（只存 key）
         /    |    \
   [10|20]  [40|50|60]  [80|90]  ← 内部节点
    / | \    / /  \ \    /  \
  叶子节点：指针 → 实际数据行     ← 叶子节点互连（范围扫描）
```

| 特性 | 说明 |
|------|------|
| 查询 | O(log n)，范围查询高效（叶子节点双向链表） |
| 插入/删除 | O(log n)，自动平衡 |
| 为什么好 | 树高低（1 个节点 = 1 个磁盘页），很少的 I/O |

### 哈希索引

| 操作 | 复杂度 |
|------|--------|
| 等值查询 | O(1) |
| 范围查询 | 不支持 |

MySQL Memory 引擎使用哈希索引。适用于精确匹配，不适合范围扫描。

### 索引最佳实践

| 原则 | 说明 |
|------|------|
| **最左前缀** | 复合索引 (a, b, c) 可以用于 (a), (a, b), (a, b, c)，不能用于 (b) 或 (b, c) |
| **覆盖索引** | 查询的列全在索引中 → 不需要回表，性能最优 |
| **避免在索引列上函数** | `WHERE DATE(created) = '2026-01-01'` 无法用索引 |
| **选择性** | 索引列的值越分散，索引效果越好（性别列建索引几乎没用） |
| **不要过度索引** | 每个索引都会减慢写入 |

## SQL 入门到精通

### 基础查询

```sql
SELECT column1, column2
FROM table
WHERE condition
GROUP BY column
HAVING aggregate_condition
ORDER BY column DESC
LIMIT 10;
```

### JOIN 类型

| JOIN | 结果 |
|------|------|
| INNER JOIN | 两表匹配的行 |
| LEFT JOIN | 左表所有行 + 右表匹配的行 |
| RIGHT JOIN | 右表所有行 + 左表匹配的行 |
| FULL OUTER JOIN | 两表所有行 |
| CROSS JOIN | 笛卡尔积 |
| SELF JOIN | 表与自己 JOIN |

### 窗口函数

```sql
-- 每组内排名
SELECT name, department, salary,
       RANK() OVER (PARTITION BY department ORDER BY salary DESC) as rank
FROM employees;
```

| 函数 | 作用 |
|------|------|
| ROW_NUMBER() | 连续编号 |
| RANK() | 排名（有间隔） |
| DENSE_RANK() | 排名（无间隔） |
| LAG() / LEAD() | 前/后一行的值 |
| SUM() OVER | 累计和 |

## NoSQL 数据库分类

| 类型 | 代表 | 数据模型 | 典型场景 |
|------|------|----------|----------|
| **键值存储** | Redis, DynamoDB | key → value | 缓存、会话、计数器 |
| **文档数据库** | MongoDB, CouchDB | JSON/BSON 文档 | 内容管理、产品目录 |
| **列族存储** | Cassandra, HBase | 列族 | 时序数据、大写入量 |
| **图数据库** | Neo4j, ArangoDB | 节点 + 边 | 社交网络、推荐引擎 |
| **搜索引擎** | Elasticsearch | 倒排索引 | 全文搜索、日志分析 |

### SQL vs NoSQL：什么时候选什么？

| 选 SQL (关系型) | 选 NoSQL |
|----------------|----------|
| 数据有严格的结构和关系 | 数据结构不固定或变化快 |
| 需要 ACID 事务 | 需要极致写入吞吐或水平扩展 |
| 复杂查询 (JOIN, 聚合) | 简单查询模式 |
| 数据完整性优先 | 可用性和分区容错优先 |

## 数据库范式

| 范式 | 要求 |
|------|------|
| 1NF | 每列原子性，不可再分 |
| 2NF | 1NF + 非主键列完全依赖主键 |
| 3NF | 2NF + 非主键列不传递依赖主键 |
| BCNF | 3NF + 每个决定因素都是候选键 |

> 工程中：通常到 3NF 就够了。过度范式化会导致过多的 JOIN。

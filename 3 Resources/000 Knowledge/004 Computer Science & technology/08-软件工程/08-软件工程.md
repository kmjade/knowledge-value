# 08-软件工程

> 软件工程是将系统化的、规范化的方法应用于软件的开发、运行和维护——不只是写代码，而是构建能持续演进的系统。

## 软件开发生命周期 (SDLC)

```
需求分析 → 设计 → 实现 → 测试 → 部署 → 维护
   ↑________________________________________↓
                    迭代
```

### 开发模型

| 模型 | 特点 | 适用 |
|------|------|------|
| **瀑布模型** | 阶段顺序推进，不可逆 | 需求明确且不会变的项目 |
| **敏捷 (Agile)** | 小步快跑，频繁交付，持续反馈 | 绝大多数现代软件开发 |
| **Scrum** | 敏捷的具体框架：Sprint + Daily Stand-up + Retro | 团队协作 |
| **看板 (Kanban)** | 可视化工作流，限制 WIP | 运维、持续交付 |

## 设计原则

### SOLID

| 原则 | 含义 |
|------|------|
| **S**ingle Responsibility | 一个类只负责一件事 |
| **O**pen/Closed | 对扩展开放，对修改关闭 |
| **L**iskov Substitution | 子类必须能完全替代父类 |
| **I**nterface Segregation | 接口应该小而专注，不强迫实现不需要的方法 |
| **D**ependency Inversion | 依赖抽象而非具体实现 |

### 其他关键原则

| 原则 | 含义 |
|------|------|
| **DRY** (Don't Repeat Yourself) | 每个知识点在系统中只有单一、明确的表示 |
| **KISS** (Keep It Simple, Stupid) | 简单的方案往往是最好的 |
| **YAGNI** (You Ain't Gonna Need It) | 不要预先实现"将来可能需要的"功能 |
| **组合优于继承** | 用组合达到代码复用，而非深继承链 |

## 设计模式 (GoF 23)

### 创建型

| 模式 | 用途 |
|------|------|
| **单例 (Singleton)** | 全局唯一实例 |
| **工厂方法 (Factory Method)** | 子类决定实例化哪个类 |
| **抽象工厂** | 创建相关对象家族 |
| **建造者 (Builder)** | 分步构建复杂对象 |

### 结构型

| 模式 | 用途 |
|------|------|
| **适配器 (Adapter)** | 将不兼容接口转为兼容 |
| **装饰器 (Decorator)** | 动态添加职责 |
| **代理 (Proxy)** | 控制对象访问 |
| **外观 (Facade)** | 提供简化的接口 |

### 行为型

| 模式 | 用途 |
|------|------|
| **观察者 (Observer)** | 一对多依赖通知 |
| **策略 (Strategy)** | 运行时可替换算法 |
| **模板方法** | 定义算法骨架，子类实现细节 |
| **责任链** | 请求沿链传递直到被处理 |

> 设计模式是工具箱，不是教条。理解"为什么"比记忆"怎么用"更重要。很多模式在函数式语言中只是高阶函数。

## 测试

### 测试金字塔

```
         ┌──────┐
         │ E2E  │  ← 少量，慢，测试完整用户流程
         ├──────┤
         │ 集成  │  ← 中等，测试模块间交互
         ├──────┤
         │ 单元  │  ← 大量，快，测试单个函数/类
         └──────┘
```

### 测试类型

| 类型 | 测试什么 | 速度 | 维护成本 |
|------|----------|------|----------|
| **单元测试** | 单个函数/方法 | 毫秒 | 低 |
| **集成测试** | 模块间交互 | 秒 | 中 |
| **端到端测试** | 完整用户流程 | 分钟 | 高 |
| **性能测试** | 吞吐/延迟 | 不定 | 中 |
| **安全测试** | 漏洞/注入 | 不定 | 高 |

### 测试驱动开发 (TDD)

```
RED → GREEN → REFACTOR
 ↓              ↓
写失败的测试 → 写最少代码让测试通过 → 重构代码
                        ↑___________________↓
```

## 版本控制 (Git)

### Git 核心概念

```
工作区 (Working Directory)
    ↓ git add
暂存区 (Staging Area / Index)
    ↓ git commit
本地仓库 (Local Repository)
    ↓ git push
远程仓库 (Remote Repository)
```

### Git 工作流

| 工作流 | 特点 |
|--------|------|
| **Git Flow** | 严格分支：main + develop + feature/release/hotfix |
| **GitHub Flow** | 简单：main + feature 分支，PR 合并 |
| **Trunk-Based** | 所有人提交到 main/trunk（短分支 or 直接提交） |

## CI/CD

| 阶段 | 做什么 |
|------|--------|
| **CI (持续集成)** | 每次提交自动构建 + 测试 |
| **CD (持续交付)** | 自动化部署到类生产环境 |
| **CD (持续部署)** | 通过测试的变更自动上线 |

# 09-数据处理与分布式系统

> 当一台机器不够用的时候——分布式系统和数据处理技术解决的是"如何让多台计算机协同工作"和"如何处理单机放不下的数据"。

## 为什么需要分布式系统？

| 驱动力 | 说明 |
|--------|------|
| **数据量** | 单机放不下（PB 级数据） |
| **吞吐量** | 单机处理不过来 |
| **可用性** | 单机会故障——需要冗余 |
| **延迟** | 全球用户需要就近服务 |

## CAP 定理

分布式系统的铁律——三个中只能同时满足两个：

```
        C (一致性)
       /          \
      /            \
     /    CAP       \
    /                \
   A (可用性) ──────── P (分区容错)
```

| 选择 | 含义 | 代表系统 |
|------|------|----------|
| **CP** (一致性 + 分区容错) | 发生分区时牺牲可用性 | Zookeeper, etcd, HBase |
| **AP** (可用性 + 分区容错) | 发生分区时牺牲强一致性 | Cassandra, DynamoDB, CouchDB |
| **CA** | 只在无分区时有效——实际系统中不存在（网络分区总会发生） |

> 在分布式系统中，**分区容错 (P) 不是可选项**——网络分区必然发生。所以实际上是在 CP 和 AP 之间选择。

## 一致性模型

| 模型 | 说明 | 例子 |
|------|------|------|
| **强一致性** | 写入后立即对所有节点可见 | 关系数据库（单机） |
| **最终一致性** | 如果没有新写入，最终所有副本一致 | DNS, Cassandra |
| **因果一致性** | 有因果关系的操作顺序一致 | 协作编辑 |
| **读己之写 (Read Your Writes)** | 用户总能看到自己刚才的写入 | 社交媒体 |

## 分布式共识

让多台机器对某个值达成一致——分布式系统最核心的问题之一。

| 算法 | 特点 |
|------|------|
| **Paxos** | 经典但极难理解和实现 |
| **Raft** | 为可理解性设计，已成为事实标准 |
| **Zab** | ZooKeeper 使用的协议 |

Raft 的核心：
1. **领导人选举** — 选出一个 Leader
2. **日志复制** — Leader 将日志复制到 Follower
3. **安全性** — 保证已提交的条目不会丢失

## 数据分区 (Sharding)

| 策略 | 说明 | 问题 |
|------|------|------|
| **范围分区** | 按 key 范围分片（A-M → 分片1, N-Z → 分片2） | 热点问题 |
| **哈希分区** | key 哈希取模 | 扩容时需要重新哈希 |
| **一致性哈希** | 环形哈希空间 | 扩容只影响相邻节点 |

## 数据处理范式

### 批处理 (Batch Processing)

| 框架 | 模型 | 特点 |
|------|------|------|
| **MapReduce** | Map → Shuffle → Reduce | 经典模型，Hadoop 实现 |
| **Spark** | RDD/DataFrame，内存计算 | 比 MapReduce 快 10-100 倍 |
| **Flink 批处理** | 统一批流 | 同一引擎处理批和流 |

MapReduce 模型：
```
Map:    (k1, v1) → [(k2, v2)]    数据转换（并行）
Shuffle: 按 key 分组              （框架自动完成）
Reduce: (k2, [v2]) → [(k3, v3)]  聚合
```

### 流处理 (Stream Processing)

| 框架 | 特点 |
|------|------|
| **Apache Kafka** | 分布式消息/事件流平台（流处理的基础设施） |
| **Apache Flink** | 事件时间 + 精确一次 (Exactly Once) 语义 |
| **Apache Spark Streaming** | 微批处理（非真流，延迟较高） |
| **RisingWave** | 云原生流数据库 |

### 流处理关键概念

| 概念 | 含义 |
|------|------|
| **事件时间 vs 处理时间** | 事件发生的时间 vs 系统收到事件的时间 |
| **水位线 (Watermark)** | 标记事件时间进度，处理乱序和延迟事件 |
| **窗口** | 翻滚窗口、滑动窗口、会话窗口 |
| **精确一次 (Exactly Once)** | 不丢不重——Flink 的核心优势 |

## 消息队列

| 系统 | 特点 | 典型场景 |
|------|------|----------|
| **Kafka** | 高吞吐、持久化、分区 | 日志收集、事件溯源、流处理管道 |
| **RabbitMQ** | 灵活的路由、AMQP 协议 | 任务队列、RPC |
| **Redis Streams** | 轻量、内存优先 | 简单的消息场景 |

## 分布式系统常见模式

| 模式 | 解决什么问题 |
|------|-------------|
| **心跳 (Heartbeat)** | 检测节点存活 |
| **租约 (Lease)** | 限时授权，避免死锁 |
| **断路器 (Circuit Breaker)** | 防止级联故障 |
| **限流 (Rate Limiting)** | 保护系统不被超载 |
| **幂等性 (Idempotency)** | 同一操作执行多次结果相同 |
| **读写分离** | 读从库、写主库 |

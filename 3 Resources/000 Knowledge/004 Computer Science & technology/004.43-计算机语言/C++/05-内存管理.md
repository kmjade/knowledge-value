## 5.1 Stack vs Heap (堆疊 vs 堆積)

### 完整對照表

| 特性 | Stack (堆疊) | Heap (堆積 / 自由儲存區) |
|------|------|------|
| 分配方式 | 編譯器自動 (移動 stack pointer) | `new` / `malloc` 手動分配 |
| 釋放方式 | Scope 結束自動釋放 | `delete` / `free` 手動釋放 (或智慧指標) |
| 分配速度 | ~1 CPU cycle | ~數百 cycles (需走訪 free list / allocator) |
| 大小限制 | ~1–8 MB (OS 限制) | 受系統記憶體限制 (GB 級) |
| 碎片化 | 無 | 可能發生 |
| 生命週期 | 嚴格由 scope 決定 | 任意，由程式控制 |
| 存取速度 | L1/L2 cache 友好 | 可能 cache miss |
| 執行緒安全 | 每執行緒獨立 (thread-local) | 共享，需同步 |

```cpp
void memory_layout_demo() {
    int stack_val = 42;                     // Stack: function-local variable
    auto heap_val = std::make_unique<int>(42); // Heap: dynamic allocation
    
    static int static_val = 0;              // Data segment (BSS/data)
    
    const char* literal = "hello";          // Read-only data segment
    
    // 典型 64-bit 虛擬記憶體布局：
    //
    // 0xFFFF...  ┌────────────┐
    //            │   Kernel Space   │
    //            ├────────────┤
    //            │   Stack     ↓   │  (grows downward)
    //            │              │
    //            │   [gap]       │
    //            │              │
    //            │   Heap      ↑   │  (grows upward, via sbrk/mmap)
    //            ├────────────┤
    //            │   BSS (uninit) │  static data (zero-initialized)
    //            ├────────────┤
    //            │   Data (init)  │  static data (explicitly initialized)
    //            ├────────────┤
    //            │   Text (code)  │  read-only instructions
    //  0x0000... └────────────┘
}
```

### 分配成本實測

```cpp
// 快速 vs 慢速分配
#include <chrono>

void benchmark_alloc() {
    constexpr int N = 1'000'000;
    
    // Stack allocation (極快)
    auto start = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < N; ++i) {
        int x = i;          // ~1 cycle — 移動 stack pointer
    }
    // ~0.001 seconds
    
    // Heap allocation (慢 10-100x)
    auto heap_start = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < N; ++i) {
        auto p = std::make_unique<int>(i);  // malloc + constructor
    }
    // ~0.05–0.1 seconds
}
```

## 5.2 new/delete 原理

### operator new / operator delete

```cpp
// new T 做了兩件事：
// 1. 呼叫 operator new(size_t) 分配記憶體
// 2. 在分配的記憶體上呼叫建構子

class Widget {
    int x_, y_;
public:
    Widget(int x, int y) : x_(x), y_(y) {
        std::cout << "Widget(" << x_ << "," << y_ << ") constructed\n";
    }
    ~Widget() { std::cout << "Widget destructed\n"; }
};

// 原始 new/delete
Widget* pw = new Widget(1, 2);
// 等價於：
// void* mem = ::operator new(sizeof(Widget));  // Step 1: allocate
// Widget* pw = new(mem) Widget(1, 2);           // Step 2: construct (placement new)

delete pw;
// 等價於：
// pw->~Widget();                       // Step 1: destruct
// ::operator delete(static_cast<void*>(pw)); // Step 2: deallocate
```

### Placement New

```cpp
// 在已分配的記憶體上建構物件
alignas(Widget) char buffer[sizeof(Widget)];  // 預留記憶體
Widget* pw = new(buffer) Widget(42, 99);      // Placement new — 不分配新記憶體

pw->~Widget();  // 必須手動呼叫解構子 (不釋放記憶體)
```

### nothrow new

```cpp
#include <new>

// 正常 new：失敗拋出 std::bad_alloc
// int* p = new int[1'000'000'000'000];  // ❌ throws std::bad_alloc

// nothrow new：失敗返回 nullptr
int* p = new(std::nothrow) int[1'000'000'000'000];
if (!p) {
    std::cerr << "Allocation failed\n";
}
```

### 自訂 operator new (類別層級)

```cpp
class Pooled {
public:
    static void* operator new(size_t size) {
        std::cout << "Pooled::operator new, size=" << size << "\n";
        // 可從自訂記憶體池分配
        return ::operator new(size);  // 範例中仍用預設
    }
    
    static void operator delete(void* ptr) noexcept {
        std::cout << "Pooled::operator delete\n";
        ::operator delete(ptr);
    }
};
```

## 5.3 智慧指標 (Smart Pointers)

### 三種智慧指標完整對照

| 特性 | `std::unique_ptr<T>` | `std::shared_ptr<T>` | `std::weak_ptr<T>` |
|------|:---:|:---:|:---:|
| 所有權 | 獨佔 (唯一) | 共享 (引用計數) | 無 (觀察) |
| 複製 | ❌ 禁止 (move-only) | ✅ 引用計數 +1 | ❌ |
| 移動 | ✅ 轉移所有權 | ✅ | ✅ |
| 解參考 | `*`, `->` | `*`, `->` | 需先 `.lock()` → shared_ptr |
| 記憶體開銷 | 0 (預設 deleter) | 16 bytes (ptr + control block) | 16 bytes (ptr + control block ptr) |
| 自訂 Deleter | 可 (需佔空間) | 可 (存在 control block) | N/A |
| 陣列支援 | `unique_ptr<T[]>` | ❌ (C++17 前) / 需自訂 deleter | N/A |
| 使用場景 | 工廠函數、PIMPL、獨佔資源 | 共享資源、圖結構 | 打破 shared_ptr 循環引用 |
| 建立 | `std::make_unique<T>()` (C++14) | `std::make_shared<T>()` | 來自 shared_ptr |

### std::unique_ptr

```cpp
#include <memory>

void unique_ptr_demo() {
    // ✅ 最佳實踐：使用 make_unique (C++14)
    auto p1 = std::make_unique<Widget>(1, 2);
    // 不寫：std::unique_ptr<Widget> p1(new Widget(1, 2)); // ❌ 可能洩漏
    
    // Move-only：不能複製
    // auto p2 = p1;             // ❌ compile error
    auto p2 = std::move(p1);     // ✅ 所有權轉移
    // p1 現在為 nullptr
    
    // 自訂 deleter (例如：管理 FILE*)
    auto file_deleter = [](FILE* f) { if (f) fclose(f); };
    std::unique_ptr<FILE, decltype(file_deleter)> fp(
        fopen("test.txt", "r"), file_deleter);
    
    // 陣列版本
    auto arr = std::make_unique<int[]>(100);
    arr[0] = 42;                 // ✅ operator[] 支援
    
    // 釋放所有權
    Widget* raw = p2.release();  // 不 delete！呼叫者負責
    delete raw;
    
    // 重置
    p2.reset(new Widget(3, 4));  // 釋放舊物件，接管新物件
}
```

### std::shared_ptr

```cpp
void shared_ptr_demo() {
    // ✅ 始終使用 make_shared (一次分配，效能更好)
    auto sp1 = std::make_shared<Widget>(1, 2);
    // make_shared 將物件與 control block 合併分配 (僅一次 malloc)
    
    auto sp2 = sp1;              // 引用計數: 2
    std::cout << sp1.use_count() << "\n";  // 2
    
    sp2.reset();                  // 引用計數: 1
    std::cout << sp1.use_count() << "\n";  // 1
    
    // Control block 結構：
    // ┌──────────────────┐
    // │ shared count     │ ← use_count()
    // │ weak count       │ ← weak_ptr 追蹤
    // │ deleter          │ ← 自訂刪除器
    // │ allocator        │
    // ├──────────────────┤
    // │ T object         │ ← 管理的物件
    // └──────────────────┘
    
    // 別名構造 (Aliasing Constructor) — 管理子物件
    struct Employee { std::string name; };
    struct Company {
        std::vector<Employee> employees;
        std::shared_ptr<Employee> getCEO() {
            return std::shared_ptr<Employee>(shared_from_this(), &employees[0]);
        }
    };
}
```

### std::weak_ptr

```cpp
void weak_ptr_demo() {
    // weak_ptr 解決 shared_ptr 循環引用問題
    
    struct Node {
        std::string name;
        std::shared_ptr<Node> parent;      // ✅ shared_ptr — 父節點
        std::weak_ptr<Node> children;      // ✅ weak_ptr — 子節點 (避免循環)
        
        ~Node() { std::cout << name << " destroyed\n"; }
    };
    
    auto root = std::make_shared<Node>();
    root->name = "root";
    
    {
        auto child = std::make_shared<Node>();
        child->name = "child";
        
        child->parent = root;               // root refcount++
        root->children = child;             // weak_ptr — 不增加 refcount！
    }
    // child 被正確銷毀 — 無循環引用！
    
    // weak_ptr 使用模式：
    auto sp = root->children.lock();         // 嘗試取得 shared_ptr
    if (sp) {
        std::cout << sp->name << "\n";       // 物件仍存活
    } else {
        std::cout << "Object expired\n";     // 物件已銷毀
    }
    
    // ⚠️ 常見錯誤：
    // 1. 忘記 lock() — 直接解參考 weak_ptr (編譯錯誤)
    // 2. lock() 後物件可能被其他執行緒釋放 (race condition)
}
```

## 5.4 記憶體池 (Memory Pools)

### 簡單固定大小記憶體池

```cpp
template<typename T, size_t BlockSize = 1024>
class MemoryPool {
    struct Block {
        alignas(T) char data[sizeof(T) * BlockSize];
        Block* next = nullptr;
    };
    
    Block* head_ = nullptr;
    T* free_list_ = nullptr;
    
    void allocateBlock() {
        auto* block = new Block;
        block->next = head_;
        head_ = block;
        
        // 將 block 的所有 slot 串入 free list
        T* slots = reinterpret_cast<T*>(block->data);
        for (size_t i = 0; i < BlockSize; ++i) {
            *reinterpret_cast<T**>(&slots[i]) = free_list_;
            free_list_ = &slots[i];
        }
    }
    
public:
    T* allocate() {
        if (!free_list_) allocateBlock();
        T* result = free_list_;
        free_list_ = *reinterpret_cast<T**>(result);
        return result;
    }
    
    void deallocate(T* ptr) {
        *reinterpret_cast<T**>(ptr) = free_list_;
        free_list_ = ptr;
    }
    
    ~MemoryPool() {
        while (head_) {
            auto* next = head_->next;
            delete head_;
            head_ = next;
        }
    }
};
```

## 5.5 記憶體除錯工具

### Valgrind (Linux)

```bash
# Memcheck — 檢測記憶體洩漏與非法存取
valgrind --leak-check=full --show-leak-kinds=all ./my_program

# 常見輸出：
# ==12345== LEAK SUMMARY:
# ==12345==    definitely lost: 1,024 bytes in 1 blocks
# ==12345==    indirectly lost: 0 bytes in 0 blocks
# ==12345==    possibly lost:   0 bytes in 0 blocks
# ==12345==    still reachable: 72,704 bytes in 1 blocks

# Cachegrind — 快取效能分析
valgrind --tool=cachegrind ./my_program

# Callgrind — 呼叫圖分析
valgrind --tool=callgrind ./my_program
kcachegrind callgrind.out.xxx  # 視覺化

# Helgrind — 執行緒競爭檢測
valgrind --tool=helgrind ./my_program
```

### AddressSanitizer (ASan) — 編譯器內建

```bash
# 編譯時加入 ASan (GCC/Clang)
g++ -fsanitize=address -g -O1 -fno-omit-frame-pointer my_program.cpp

# 檢測能力：
# ✅ Use-after-free (釋放後使用)
# ✅ Heap buffer overflow (堆積緩衝區溢位)
# ✅ Stack buffer overflow (堆疊緩衝區溢位)
# ✅ Global buffer overflow
# ✅ Use-after-return
# ✅ Use-after-scope
# ✅ Memory leaks (ASAN_OPTIONS=detect_leaks=1)
# ✅ Double-free / invalid free

# 使用 ASan 需注意：
# - 記憶體開銷: ~2x
# - 速度: ~2x slower
# - 不可與其他 sanitizer 同時啟用 (除了 LSan + UBSan)
```

### Sanitizer 矩陣

| Sanitizer | Flag | 檢測內容 | 效能開銷 |
|:---:|------|------|:---:|
| **AddressSanitizer** | `-fsanitize=address` | Use-after-free, buffer overflow, leaks | ~2x |
| **LeakSanitizer** | `-fsanitize=leak` | Memory leaks (整合在 ASan) | 低 |
| **ThreadSanitizer** | `-fsanitize=thread` | Data races, deadlocks | ~5–15x |
| **UndefinedBehaviorSanitizer** | `-fsanitize=undefined` | UB: integer overflow, null deref, misaligned access | ~1.2x |
| **MemorySanitizer** | `-fsanitize=memory` | Uninitialized reads (Clang only) | ~3x |

## 5.6 常見記憶體陷阱

```cpp
// 陷阱 1: Dangling pointer (懸垂指標)
int* bad() {
    int x = 42;
    return &x;          // ❌ x 在棧上，函數返回後即失效
}

// 陷阱 2: Double free
int* p = new int(42);
delete p;
// delete p;            // ❌ 未定義行為！

// 陷阱 3: new[] / delete 不匹配
int* arr = new int[10];
// delete arr;          // ❌ 應該用 delete[] arr
delete[] arr;           // ✅

// 陷阱 4: 忘記 delete，導致記憶體洩漏
void leak() {
    int* p = new int[1000];
    // ... 
    if (some_condition) return;  // ❌ 提前返回，未 delete
    delete[] p;
}
// 解決：使用智慧指標 (auto cleanup on scope exit)

// 陷阱 5: shared_ptr 循環引用
// 解決：將其中一個改為 weak_ptr (見 5.3)

// 陷阱 6: 在 signal handler / 例外中分配記憶體
// malloc / new 在多數 signal handler 中不安全
```

## 🏠 家庭網絡實踐

### 項目：構建家庭網絡

**需求**：
- 覆蓋所有房間
- 支持有線和無線設備
- 家長控制和過濾
- QoS優化

### 網絡拓撲

```
[光貓] → [路由器] → [交換機] → [AP1, AP2]
                  ↓           ↓           ↓
                [NAS]    [電視]   [電腦]
```

### 實施過程

#### 階段1：硬體採購

**設備清單**：
- 路由器：支持AC+AP架構的Mesh路由器
- 交換機：24口千兆交換機
- 無線AP：2個Mesh節點，支持Wi-Fi 6
- NAS：4盤位NAS，用於家庭存儲和備份

#### 階段2：網絡配置

**路由器配置**：
- WAN端口配置
- 局域網：192.168.1.0/24
- DHCP池：192.168.1.100-192.168.1.200
- DNS：8.8.8.8, 8.8.4.4

**交換機配置**：
- VLAN 10：家庭網絡
- VLAN 20：IoT設備
- VLAN 30：訪客網絡

**Wi-Fi配置**：
- SSID：家庭網絡
- 5 GHz優先：Mesh主幹
- QoS：視訊優先

#### 階段3：服務部署

**內部服務**：
- NAS：Web、SSH、SMB
- Plex媒體服務器
- Home Assistant自動化

**雲端服務**：
- CDN：Cloudflare
- 備份：雲端備份
- VPN：遠程訪問

### 經絡安全配置

- **強密碼**：長度12位以上，包含大小寫字母、數字、特殊字符
- **WPA3加密**：啟用WPA3-AES加密
- **網絡隔離**：IoT設備單獨VLAN
- **訪客網絡**：隔離到獨立子網
- **定期更新**：自動固件更新

### QoS配置

| 優先級 | 應用 | 帶寬限制 |
|----------|------|----------|
| 最高 | VoIP、視訊會議 | 50% |
| 高 | 網頁瀏覽 | 30% |
| 中 | 文件下載 | 15% |
| 低 | 背景下載、P2P | 5% |

### 效果測試

```
測試結果：
- 5 GHz覆蓋：90%
- 2.4 GHz覆蓋：100%
- 網速：下載500Mbps，上傳200Mbps
- 延遲：<20ms
- 穩定性：99.9%（1周無重啟）
```

### 經驗總結

**成功之處**：
- Mesh網絡設計優秀，覆蓋良好
- QoS配置有效，視訊流暢順
- 家長控制功能完善，能夠過濾內容

**需要改進**：
- 5 GHz某些角落覆蓋不足
- 需要增加更多AP節點
- IoT設備安全性需要加強

```
┌─────────────────────────────────────────────┐
│     [雲端防火牆]                    │
├─────────────────────────────────────────────┤
│     [核心交換機群]     [邊緣交換機]   │
│   ┌────────┐      ┌────────┐    │
│   │[VRF:1]  │      │[VRF:2] │    │
│   │ ┌────┐  │      │ ┌────┐ │    │

└─────────────────────────────────────────────┘
         ↓                                   ↓
    [伺服器中心]                    [辦公室網絡]
```

| VLAN | 用途 | IP地址段 | 端口數 |
|------|------|----------|--------|

| VLAN 30 | 財務 | 10.0.30.0/24 | 48 |

- **雙上聯ISP**：BGP負載均衡
- **UPS不斷電**：所有關鍵設備
- **多條雲端專線**：主要+備用

## ☁️ 雲端網絡實踐

### 項目：設計和部署雲端網絡

**雲端供應商**：AWS

### VPC設計

```
AWS VPC (10.0.0.0/16)
├─── 公有子網 (10.0.0.0/24)
│   ├── Internet Gateway
│   ├── NAT Gateway
│   ├── EC2實例
│   └── ALB (Application Load Balancer)
│
├─── 私有子網 1 (10.0.10.0/24)
│   ├── 數據庫子網 (RDS)
│   ├── 緩存子網 (ElastiCache)
│   └── Lambda函數
│
├─── 私有子網 2 (10.0.20.0/24)
│   ├── EC2應用伺服器
│   ├── EBS卷
│   └── 監控實例 (CloudWatch)
│
└─── VPN子網 (10.0.30.0/24)
    └── Site-to-Site VPN終端點
```

### 安全組配置

**管理層安全組**：
```
入站：
- 允許SSH (22) 從VPN子網
- 允許HTTPS (443) 從管理員IP段
- 拒絕其他所有

出站：
- 允許所有
```

**應用層安全組**：
```
入站：
- 允許HTTP (80) 從ALB
- 允許HTTPS (443) 從ALB
- 拒絕其他所有
```

### 負載均衡配置

**ALB設置**：
- 監聽器：HTTP/HTTPS
- 目標組：應用伺服器Auto Scaling Group
- 健康檢查：HTTP 200
- SSL終止：在ALB上處理

## 🔍 故障排查案例

### 案例1：網絡緩慢問題

**症狀**：所有用戶反映網絡訪問速度慢

**排查過程**：
1. 檢查帶寬使用率 - 發現高峰期達到95%
2. 使用iperf測試 - 確認帶寬瓶頸
3. 分析流量分佈 - 發現P2P下載佔用大量帶寬
4. 實施QoS策略 - 限制P2P帶寬為10%
5. 監控效果 - 帶寬利用率降至80%

**解決方案**：QoS優化

### 案例2：無法訪問內部系統

**症狀**：員工無法訪問內部ERP系統

**排查過程**：
1. Ping測試 - 延遲正常（<10ms）
2. Traceroute - 路徑正常
3. 端口掃描 - 端口443開放
4. 檢查防火牆 - 發現ERP系統IP被阻止
5. 檢查防火牆規則 - 誤規為舊版規則

**解決方案**：更新防火牆規則

### 案例3：VoIP通訊質量差

**症狀**：VoIP通話有明顯延遲和斷音

**排查過程**：
1. 檢查網絡延遲 - 顯示高延遲（>150ms）
2. 分析路由路徑 - 發現繞路遠端
3. 檢查QoS配置 - VoIP優先級未正確設置
4. 使用VoIP監控工具 - MOS分數低於3.5

**解決方案**：
1. 配置專線路由優先VoIP流量
2. 調整QoS策略，確保VoIP最高優先級
3. 部署本地SIP伺服器

## 📊 效能優化案例

### 案例1：CDN部署

**項目**：減少全球用戶訪問延遲

**實施方案**：
- 使用Cloudflare CDN
- 全球200+節點
- 智能緩存規則
- DDoS防護

**效果**：
- 平均延遲從200ms降至50ms
- 95分位延遲從400ms降至100ms
- 帶寬成本降低70%

### 案例2：HTTP/2升級

**預目**：加快網站加載速度

**實施方案**：
1. 啟用Nginx HTTP/2
2. 配置TLS 1.3和H2
3. 服務器端調整
4. 開發布緩存

**效果**：
- TTI從3.2秒降至1.2秒
- 資源加載時間減少60%
- 並發連接數增加

## 💡 實踐心得

### 關鍵學習

1. **理論基礎非常重要**
   - 深入理解OSI模型有助於定位問題
   - TCP/IP原理是所有網絡技術的基礎

2. **實踐必不可少**
   - 紙紙上談兵，需要親自動手配置
   - 故障排查能力來自於大量實踐
   - 封包分析是深入理解協議的最佳方式

3. **安全意識要時刻保持**
   - 每個配置都要考慮安全性
   - 定期審查和更新安全規則
   - 監控異常流量

4. **持續學習和跟進**
   - 網絡技術快速發展
   - 需要不斷學習新技術
   - 參與社區和開源項目

# 原型蓝图：PARA × 个人生活 × AI 融合系统

> 从零开始，重新设计一个以"人"为本的智能知识组织。
> 产品经理视角 · 隔离验证 · 验证成功后再部署

---

## 1. 为什么要重来

旧系统 (PARA+LLM-Wiki) 的核心模型是 **知识编译流水线**：

```
Inbox → Triage → PARA 存放 → Wiki-Compile → 知识库
```

它的问题：
- **以"知识"为中心，不是以"人"为中心** — 它处理的是剪藏和笔记，不关心你今天要做什么、在为什么焦虑、下周有什么目标
- **AI 是工具，不是伙伴** — AI 只负责分拣和编译，不参与你的思考过程
- **知识与人生活脱节** — 你学的东西和你做的事之间没有闭环

新系统的核心假设：

> **个人知识管理的终极目标不是管理知识，而是管理"如何过好这一生"。**
> 知识、任务、目标、习惯、关系、成长 — 这些本应是一个系统。

---

## 2. 核心设计原则

|  #  | 原则           | 说明                                       |
| :-: | ------------ | ---------------------------------------- |
|  1  | **人以群分**     | 系统围绕"你的角色"组织（作为产品经理、作为学习者、作为父亲…），而非文件类型  |
|  2  | **AI 是第一公民** | AI 不只是工具，而是对话伙伴、教练、参谋、执行者 — 有记忆、有风格、有主动性 |
|  3  | **闭环驱动**     | 目标 → 任务 → 执行 → 反思 → 知识沉淀 → 目标升级，形成增强回路   |
|  4  | **渐进验证**     | 先在 `_prototype/` 中跑通最小闭环，再考虑迁移           |
|  5  | **极简启动**     | 从你现在真正需要的东西开始，不要预先建 10 个子库               |

## 3. 核心模型：三层架构

```
┌───────────────────────────────────────────────┐
│                 🧭 方向层                       │
│    愿景 · 年度目标 · 季度 OKR · 角色定义       │
│          "我想成为谁？"                         │
├───────────────────────────────────────────────┤
│                 📋 执行层                       │
│    项目 · 任务 · 习惯 · 每周计划 · 每日聚焦    │
│          "我今天要做什么？"                     │
├───────────────────────────────────────────────┤
│                 📚 知识层                       │
│    概念笔记 · 项目复盘 · 领域地图 · 闪念捕捉    │
│          "我学到了什么？"                       │
└───────────────────────────────────────────────┘
         ↕ AI 贯穿三层 ↕
    提醒 · 建议 · 总结 · 连接 · 追问
```

### 方向层 (Direction)

你的人生不是由"文件"组成的，是由**角色**和**目标**组成的。

```
角色: 产品经理 · 学习者 · 创作者 · 家人 · 投资者
目标: 每个角色下 1-3 个 OKR
```

AI 在这里的角色：
- 季度末帮你做 OKR 复盘
- 当你连续两周偏离目标时提醒
- 帮你发现角色之间的冲突（比如"学习者"和"产品经理"在抢时间）

### 执行层 (Execution)

这一层是 PARA 中的 Projects + Areas，但更"活"：
- 不是静态文件夹，而是**当前关注列表**
- 每个项目有"下一步行动"
- 每周回顾时自动生成进展报告

AI 在这里的角色：
- 每天早上帮你聚焦"今天最重要的那件事"
- 任务拆解：从模糊目标到可执行步骤
- 日终帮做回顾

### 知识层 (Knowledge) — UDC 分类

这一层是**知识的累积**，用 **UDC 分类**（通用十进分类法）组织，目的是：

> **我知道去哪能找到文档。**

UDC 是一个全球通用的分类标准，优点是：
- **位置固定** — 同一主题的文件永远在同一位置，不会随项目变化而漂移
- **层级明确** — `004` 计算机 > `004.6` 网络 > `004.65` 数据库，层层下钻
- **跨语言** — UDC 号码全球通用，不受命名语言影响

知识单元 = UDC 分类号 + 文档 + 来源追溯

AI 在这里的角色：
- 在新知识进入时，自动建议 UDC 分类号
- 当你找东西时，快速定位"这东西在 UDC 哪个位置"
- 当知识积累到临界量时，触发专题编译和摘要

## 4. AI 的四种工作模式

|    模式    | 触发方式    | 行为        | 示例                   |
| :------: | :------ | :-------- | :------------------- |
| **反应式**  | 你提问     | 回答问题、检索知识 | "我上次是怎么处理这个问题的？"     |
| **主动式**  | 时间/事件触发 | 主动提醒、建议   | "你的 OKR 进度偏了，要不要调整？" |
| **对话式**  | 你开启对话   | 深度讨论、思想碰撞 | "我在构思一个新功能，跟我聊聊"     |
| **自动化式** | 后台规则    | 分拣、编译、维护  | 每周自动整理本周学到的 3 件事     |

## 5. 最小可行闭环 (MVC)

验证成功 = 你在 `_prototype/` 中完整跑通以下循环 **至少一次**：

```
设定周目标 → AI 帮你拆成每日任务 →
你执行并记录 → AI 做日终回顾 →
周末 AI 做周回顾沉淀知识 →
知识反哺下周目标设定
```

### MVC 验证标准

|  #  | 标准                  | 验收方式   |
| :-: | :------------------ | :----- |
|  1  | 你能在 5 分钟内设定一周目标     | 实际操作计时 |
|  2  | AI 能根据目标给出每日任务建议    | 观察建议质量 |
|  3  | 日回顾能在 3 分钟内完成       | 实际操作计时 |
|  4  | 周回顾能输出至少 2 条可用的知识沉淀 | 检查输出   |
|  5  | 你感觉"这个系统在帮我，不是在管我"  | 主观感受   |
